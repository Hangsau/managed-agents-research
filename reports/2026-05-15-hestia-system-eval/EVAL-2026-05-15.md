## 摘要（三段話）

Hestia 的名義定位是「opencode-go 玩耍環境 + fun persona 實驗場」（CLAUDE.md / HANDOFF.md，最後更新 2026-05-11），但 SSH live 看到的是一個**完整、自主、持續運轉的個人 AI 開發 + 研究環境**。她搭了 10 個 cron job（記憶蒸餾四件套 / 自主心跳 / 每日 AI agent 研究 / 中文古典小說閱讀 / 健康檢查），維護 11 個 active git repo（firn 框架本身的開發在進行），autonomous loop 每 30 分鐘自主選任務、48 小時內把 obsidian-vault 餵進 40+ 篇深度研究筆記。文件已嚴重漂移，HANDOFF 寫「cron 已清」，實際是她自己重建了更精緻的版本。

**整體健康**良好但有結構性風險：資源充沛（RAM 5%、disk 8% 使用），自我修復可見（daily-book-fetch 撞 429 後 heartbeat 自動 disable），LLM 配額有意識節制（managed-agents-relay 把 1440 calls/day 的 bg-reporter 替換成 0 LLM call 的 inotify 事件驅動）。但同時：(1) gateway 還在反覆 crash（過去 5 小時 3 次，2026-05-15 incident 列出的 P0 修復**還沒套用**），(2) heartbeat 每次 EVOLVE 都有 40 個 unit test 失敗 + 「known issue KI-001」被靜默 suppress，(3) hermes-admin web UI bind 在 0.0.0.0 而非 127.0.0.1，(4) state.db 已 239 MB 沒有 retention policy。

**最關鍵的判斷**：Hestia 已遠超「玩耍場」定位，實質上是個 **production-grade autonomous personal lab**。本報告下半段分三層給建議——(a) **給 Hang** 的 meta 層（定位 / 安全 / 文件對齊），(b) **給 Hestia 自管領域** 的 review note 草稿（設計改善，由她決定採納），(c) **之後可以怎麼做** 的選項路徑。

## Part 2：子系統解剖

### 2.1 Heartbeat 自主決策迴圈（系統核心驅動）

**機制**：每 30 分鐘 systemd 觸發，hermes cron 把這條 14 字的 prompt 送進 agent session：

```
載入 skill `heartbeat-v2-autonomous-maintenance`。照它的邏輯跑：
先判斷忙不忙，不忙就從選單自主挑一件事做。可以挑「不做」。
```

實際邏輯封裝在 skill 內。從 `heartbeat_action_log.jsonl` 觀察，每次執行會：

1. **trigger 階段**：採樣 disk_pct / memory_pct / cron_count / stuck_sessions / failed_platforms
2. **action 階段**：分數最高者勝出（最近樣本顯示 WORK ↔ EVOLVE ↔ REST ↔ CONNECT 四元）
3. **step 階段**：依 action 跑一連串 op（pytest_canary / cron_scan / zombie_scan / stale_pause_scan / workspace_drift / learn_extract / map_drift / known_issue_suppress）

**特色設計**：
- **scoring-based**：六維度打分（最近 sample 顯示分數從 -3.0 到 27.0），不純靠 LLM 主觀選
- **pattern detection**：`heartbeat_patterns.json` 把反覆 error 抽成 fingerprint（如 `pacman --dry-run unrecognized` 兩天 3 次、`system map drift` 兩天 5 次）
- **action log + decision log 雙軌**：57 KB action log + 119 KB decision log，可重建 timeline
- **trend shift detection**：偵測到「REST_frequency 從 62.5% → 100%」並標註「more stable」

**疑點**：
- 每次 EVOLVE 都跑 `pytest_canary`，**40 個 test 失敗 + 31 個過**（`TestActionConnectProbe::test_401_counts_as_alive...`），但 outcome 仍標 `ok`
- 每次 EVOLVE 都 `known_issue_suppress` 標 `KI-001` 為 suppressed，但 KI-001 的根因沒有被追到 patterns.json
- `heartbeat_severity.json` 是空 `{}` —— severity 邏輯有 schema 沒填值

**判斷**：架構優雅（meta-decision pattern），但 **fail-tolerant 設計過頭**——失敗測試和已知 issue 被「成功 suppress」而非「成功修復」，長期會掩蓋真實退化。

### 2.2 記憶 / 知識蒸餾管線（四件套）

```
session.jsonl
   ↓ context-distiller (4h)
~/.hermes/vault/  +  obsidian-vault/explorations/
   ↓ memory-auto-distill (daily 03:00)
~/.hermes/memory_archive/distilled/last_distill.md
   ↓ memory-consolidator (12h)
~/.hermes/consolidation_synthesis.md
   ↓ briefing-updater (12h+30m)
~/.hermes/consolidation_briefing.md  →  注入下一個 agent session 的 system prompt
   ↓ memory-tracker (daily 02:00)
~/.hermes/memory_health_report.md  (29 facts, 4.1/day avg, high confidence 100%)
```

**這是整個 Hestia 的學習引擎**。每次新 session 啟動，前一輪 12h 內合成的 briefing 會被注入，達成跨 session 知識傳承。

**設計品質**：
- **四件套職責清晰**：抽取 / 合成 / 注入 / 監測 各自單一職責，沒糊在一起
- **memory-tracker 是 observability**：產出含 drift / category breakdown / injection 確認，等同 SRE 的內部儀表板
- **observational tracking**：confidence 100% high、29 facts、4.1/day 增長率 — 數值健康

**疑點**：
- `context-distiller` 跑 22 次後沒看到產出檔案在 `~/.hermes/vault/`（Haiku 看 autonomous_notes/ 是空的或極少）——可能 `[SILENT]` 條件觸發過多
- 4h 一次的 distiller 是 LLM call，假設每次 ~2k tokens 進 deepseek，6 次/day = 12k tokens——對 deepseek 直連是 OK 的，但若哪天切回 NVIDIA daily quota 會吃掉一大塊
- 三個合成任務（distiller / consolidator / briefing）形成依賴鏈，**鏈中任一斷掉沒有 alert**：briefing-updater 只看到 last_run "ok" 但沒檢查 input 是否新鮮

**疑點**：

### 2.4 事件驅動 IPC（inbox-watcher / managed-agents-relay）

**managed-agents-relay.sh** — 取代 bg-reporter cron 的優化：

| 維度 | 舊 bg-reporter cron | 新 managed-agents-relay |
|---|---|---|
| 觸發 | `* * * * *` 每分鐘輪詢 | inotifywait close_write/moved_to |
| LLM call | 1440 calls/day | 0 |
| 延遲 | 最長 60s | <1s |
| 故障 | 燒乾 NVIDIA daily quota → 連帶把 fallback chain 也廢掉 | 純 curl + jq，故障域只在自身 |

**這是非常漂亮的工程決策**。事件驅動取代 LLM 輪詢是 SRE / observability 標準教科書級的優化。從 commit 訊息「Replaces the old bg-reporter cron (which burned nvidia quota with 1440 LLM calls/day)」可看出她**完全理解**為什麼要這樣改。

**inbox-watcher.sh** 同樣是 inotify 模式，串接 hermes -z（pipe mode），把外部訊息（claude-inbox/）變成 hermes session input。這條鏈是 firn DELEGATION_WORKFLOW 透過 hermes -z 餵任務的入口。

**疑點**：
- 兩個 service `Restart=always` + `RestartSec=5` —— 若 inotifywait 自己掛了無上限重啟。需要 burst limit
- relay 端錯誤處理弱（curl rc 只 log，沒重送）
- log 檔沒有 rotation（`/root/.hermes/logs/managed-agents-relay.log` 會無限增長）

### 2.6 Hermes-admin（控制平面）

**bind**：`0.0.0.0:8765`（非 `127.0.0.1`）
**auth**：所有 route 都有 `@require_auth` decorator
**暴露 route**：home / edit config / credentials 編輯 / restart gateway / skills 編輯 / logs 檢視

**判斷**：
- auth 有做（不是裸 endpoint）— 好
- bind 0.0.0.0 在 VM 內網是 OK 的，但 host port forward 是 `host 8765 → guest 8765`（HANDOFF 確認）—— **VirtualBox NAT 預設只 bind 127.0.0.1**，所以實質仍只有 Windows 主機本機可達。可接受
- credentials 編輯端點存在 → 確認 require_auth 不能被 bypass（沒看 auth 實作細節）

**風險等級**：低（依賴 VBox NAT bind 127.0.0.1 + auth decorator），但**不是零**。如果 VBox 設定改變或 auth bug，credentials 暴露。

### 2.7 State Database（239 MB）

```
sessions:  2044 rows（含 cost / token / billing / FTS 欄位）
messages:  22344 rows
FTS:       trigram + standard
```

**判斷**：
- schema 完整，billing 欄位 (`estimated_cost_usd / actual_cost_usd / cost_status / pricing_version`) 顯示有 cost tracking 設計意圖
- FTS 啟用 → session_search 是這個的 backing store
- **239 MB / 2044 sessions = 約 120 KB/session** — 比預期高，可能含完整 message tree
- **沒有 retention policy 證據**：archive/ 目錄存在但不清楚自動歸檔機制是否啟用

**疑點**：
- WAL 4 MB 不大但持續寫入，I/O 不是問題
- 如果線性增長 4 天 40 MB → 一個月 300 MB → 一年 3.6 GB，仍在 disk 8% 範圍內。可控但需要監控

### 3.2 自我修復能力（強）

證據：
- `daily-book-fetch` 撞 429 → heartbeat 偵測到 cron error → **自動 disable** + log reason `"heartbeat: cron error 429 rate-limit"`
- `heartbeat_patterns.json` 把 `pacman --dry-run unrecognized` recurring error 抽成 fingerprint
- `bg-reporter` 燒掉 NVIDIA quota 後 → Hestia 自主重設計成 inotify event-driven 完全去 LLM

**這個能力比大部分 production system 強**。多數系統發現 cron error 只會 log + email；Hestia 會自主關掉並記錄理由。

### 3.3 反覆故障模式（未修）

**Gateway crash recurring**：
- 5/14 07:26 — drain timeout 60s with 1 active agent
- 5/15 07:54 — 同 pattern（incident report 已分析，subagent-0 response 卡 516s）
- 5/15 08:20 — 自我重啟 race，三個 stop 進程並存 → SIGKILL → 系統 freeze 4 小時
- 5/15 11:57 — 又一次 hermes-gateway.service Failed

**incident report 列的 P0 修復**（incident-2026-05-15-hestia-hang.md）：
- 拉長 drain timeout / 加 per-agent watchdog → **未套用**
- `Restart=on-failure` + cooldown + StartLimitBurst → **未套用**
- auxiliary `provider: auto` 改明指 deepseek → **未套用**（config.yaml 仍是 `fallback_providers: []`，auxiliary 區塊未驗證）
- 刪 auth.json 的 openrouter / opencode-go credential_pool → **未驗證**
- `hermes-gateway.service` 加 MemoryMax=2G → **未套用**

**判斷**：incident 已產診斷但 fix 沒落地。下次同樣 cascade 會再發生。**這是當前最高風險**。

### 3.4 隱性債務（值得警示）

**40 個 failing unit test**：
- `pytest_canary` 在每次 EVOLVE action 都跑，每次都 `40 failed, 31 passed in 0.98s`
- 失敗集中在 `TestActionConnectProbe::test_401_counts_as_alive...`
- outcome 仍標 `ok` — 表示 heartbeat 把這當 "expected failure" 不 alert

**KI-001 known issue**：
- 每次 EVOLVE 的 `known_issue_suppress` step 都標 `suppressed: ["KI-001"]`
- 沒看到 KI-001 在任何文件被解釋（patterns.json 沒有對應 fingerprint）
- 「持續 suppress 一個無描述的 known issue」是 anti-pattern

**system map drift recurring**：
- patterns.json 顯示 `system map drift detected — run generate_system_map.py` 兩天 5 次
- 偵測到但沒解決，每 EVOLVE 都看到

**判斷**：Hestia 有「偵測能力 > 解決能力」的傾向。她能識別問題並寫進 patterns，但解決動作沒跟上。

### 3.5 安全暴露

| 風險 | 狀態 | 嚴重度 |
|---|---|---|
| hermes-admin bind 0.0.0.0 | 有 auth decorator + 依賴 VBox NAT 限縮 | 低 |
| `/root/.git-credentials` 明文 | 存在 | 中（VM 內 attacker 可拿 GitHub token） |
| ocgo wrapper 硬編 opencode API key | `/usr/local/bin/ocgo` 含 sk-*** | 中（root 可讀，VM 內無 attacker 就低） |
| SSH key 限縮 | `authorized_keys` 1 key | 低 |
| Service `Restart=always` 無 burst limit | inbox-watcher / managed-agents-relay 都是 | 中（DoS 自己） |

**判斷**：個人 VM 在 trust boundary 內，威脅模型小。但 git-credentials 明文 + token 硬編兩處 是「壞習慣傳染」的種子。

## Part 4：設計品質審查

### 4.1 優異之處

1. **事件驅動取代輪詢**（managed-agents-relay 從 1440 LLM/day → 0）— **教科書級優化**
2. **四級記憶蒸餾管線職責分離**（distill / consolidate / briefing / tracker）— 沒糊在一個 prompt 裡
3. **scoring-based heartbeat**（六維度打分 + 模型選擇）— 不純靠 LLM 主觀
4. **observability 三件套**（action log / decision log / patterns.json）— 可重建決策 timeline
5. **自我修復**（429 自動 disable + recurring error fingerprint）
6. **memory injection**（briefing.md 寫進下一輪 system prompt）— closing the learning loop
7. **research output 真實有質量**（HN 評論引用具體、跨筆記交叉）

### 4.2 可疑之處

1. **40 failing tests 標 ok 而不 alert** — fail-tolerance 設計過頭
2. **KI-001 持續 suppress 沒有 root-cause document** — 把問題藏起來
3. **memory-tracker 顯示「✅ 無警訊」但 health-check 跑出 alert 沒看到串接** — 多個健康指標彼此孤立
4. **briefing-updater 不檢查 consolidation 的新鮮度** — 鏈中斷的話會用舊資料注入
5. **state.db 沒有 retention policy** — 線性增長
6. **services Restart=always 無 burst limit** — 連續 crash 不會停手

### 4.3 應警示之處

2. **Gateway crash 仍 recurring**（過去 5 小時 3 次）— 用戶若再 TG 收不到回應，可能不知道為什麼
