# Statewright: Visual State Machines as Agent Guardrails — Deterministic Enforcement Beats Bigger Models

**Source**: https://github.com/statewright/statewright (HN launch 2026-05-13)
**Type**: repo (Apache 2.0, Rust engine + MCP plugin)
**Confidence**: medium (solid engineering + research results, but on a 5-task SWE-bench subset, not full benchmark; product freshly launched)

## 1. The Problem

AI coding agents are powerful but brittle. Give a model 40+ tools and an open-ended problem and it barely gets out of the gate. The common response: bigger models, longer system prompts. Sometimes it helps. Sometimes you get "read-loop death spirals" — the model reads the same file 5+ times without ever editing because the tool space is too large to reason about coherently.

Observability tells you what went wrong _after_ the fact. It doesn't prevent it. What's missing is a structural guarantee — a way to say "in this phase, you may only use these tools" and have it enforced at the protocol layer, not through prompt engineering.

## 2. Core Mechanism

Statewright introduces **deterministic state machine guardrails** for AI agents. The core is a Rust engine (`crates/engine`) that evaluates state machine definitions with zero LLM involvement. On top sits an MCP plugin layer that gates tool access at the protocol level.

### How it works, step by step

**Workflow definition** (JSON, generated by the agent itself via `statewright_create_workflow`):
```json
{
  "id": "bugfix",
  "initial": "planning",
  "states": {
    "planning": {
      "allowed_tools": ["Read", "Grep", "Glob"],
      "max_iterations": 8,
      "on": { "READY": "implementing" }
    },
    "implementing": {
      "allowed_tools": ["Read", "Edit", "Write"],
      "max_edit_lines": 20,
      "max_files_per_state": 3,
      "on": { "DONE": "testing" }
    },
    "testing": {
      "allowed_tools": ["Read", "Bash"],
      "allowed_commands": ["pytest", "cargo test", "npm test"],
      "on": {
        "PASS": { "target": "completed", "guard": "tests_passed" },
        "FAIL_TEST": "implementing"
      }
    }
  }
}
```

**Enforcement mechanism**: When a workflow activates, MCP hooks intercept tool calls _before the model sees them_. If a tool isn't in the current state's `allowed_tools`, the call is rejected with an error message telling the agent what IS available and how to transition. This is **hard enforcement** — unlike prompt-based approaches, the model can't jailbreak it.

**Key guardrails** (from the engine source, `crates/engine/src/`):

| Guardrail | Implementation |
|-----------|---------------|
| Per-state tool enforcement | Tools invisible to agent when not in `allowed_tools` |
| Bash discernment | Redirects (`>>`), destructive ops (`rm`, `shred`), scripting interpreters blocked in non-write states |
| Edit guards | Rejects diffs exceeding `max_edit_lines`, caps files edited per state |
| Command allow-lists | Prefix-matched `allowed_commands` per state |
| Conditional transitions | Guards with programmatic predicates (`eq`, `gt`, `exists`, `in`, `contains`) on context data |
| Approval gates | `requires_approval` pauses for human review before high-risk transitions |
| Sub-machine invocation | Nested workflows (`Invoke` transition type) — delegate to a sub-machine, resume at `on_complete` |
| Safe-next fallback | When model emits unrecognized transition event, fallback to declared `safe_next` instead of error |

**The "huh that's clever" insight**: The state machine engine is entirely deterministic — no LLM in the loop for enforcement. This means:
- It works identically on frontier and local models
- It can't be prompt-engineered around
- It's auditable (every transition logged)
- It reduces the cognitive load: the model sees 5 tools instead of 30

### Research results

On a 5-task SWE-bench subset, two small models (gpt-oss:20b, gemma4:31b) went **from 2/10 to 10/10** with statewright constraints. Same tasks, same hardware. Below ~13GB, models can produce tool calls but can't retain enough file content for accurate edits — that's the hardware floor, not a statewright limitation.

The structural win on frontier models is different: breaking read-loop death spirals and keeping the tool space small enough that the model actually reasons instead of flailing.

## 3. Why It Matters

This is a fundamentally different approach to agent reliability. Instead of treating reliability as a model-capability problem (get a smarter model), Statewright treats it as a **constraint-design problem** (make each step's context small enough that even a modest model can handle it).

Three implications:

1. **Model-agnostic reliability layer**: The same state machine works across Claude, GPT, and local models. This is the opposite of prompt-tuning per model — it's structural.

2. **Enforcement vs. advice**: Most "guardrails" today are advisory (injected into prompts). Statewright blocks tool calls at the protocol layer. The model literally cannot call a tool that isn't in the current state's allowlist.

3. **The sub-machine pattern**: `Invoke` transitions let you compose workflows hierarchically. A "deploy" workflow can invoke a "run-tests" sub-machine, then resume based on `on_complete` or `on_fail`. This enables reusable, testable sub-workflows.

## 4. Limitations / Honest Assessment

- **Small benchmark**: The 2/10→10/10 result is on a 5-task SWE-bench subset, not the full 2,294-instance benchmark. Extrapolate cautiously.
- **MCP dependency**: Hard enforcement requires MCP support in the agent. Cursor enforcement is advisory only (MCP alone can't gate tool calls in Cursor's architecture).
- **Workflow design burden**: Someone has to define the state machine. The agent can generate one via `statewright_create_workflow`, but validating and tuning it requires human judgment.
- **Over-constraint risk**: If a workflow is too restrictive, the agent gets stuck. The escape hatch (`statewright_deactivate`) exists, but you lose enforcement.
- **Rigidity vs. flexibility**: State machines work beautifully for structured tasks (bugfix, deploy, code review). They're awkward for open-ended exploration where the tool set genuinely needs to be wide.
- **Product maturity**: Launched today. Pricing, uptime, and long-term support unknown.
- **License nuance**: Core engine is Apache 2.0, but full-stack self-hosting is FSL (converts to Apache 2.0 in 2029).

## 5. Actionable for Our Projects

### firn

firn's architecture already has phase-like concepts (TaskService, Dispatcher, Watchdog) but tool access isn't gated per phase. Direct applications:

- **Phase-gated tool access in Dispatcher**: firn's Dispatcher could adopt Statewright-style per-phase tool constraints. When a task is in "analysis" phase, only read tools are available. "Execution" phase unlocks write tools. This is enforceable in firn because firn controls its own tool registry.
- **Bash discernment**: firn's shell execution could adopt the redirect/destructive-op blocking pattern. In "analysis" states, `>` and `>>` are blocked.
- **Watchdog integration**: The Watchdog could enforce `max_iterations` per state and transition to `safe_next` on timeout, rather than failing hard.
- **Sub-task invocation**: firn's TaskService could adopt the `Invoke` pattern — spawn a sub-task (run its own state machine), then resume the parent based on outcome.

### agent-infra

agent-infra's batch task runner is the most natural fit:

- **Per-task-type state machines**: Each playbook task type (research, code-review, deployment) gets a state machine definition. The runner enforces tool access per phase.
- **Research pipeline specifically**: A "scout→filter→deep-dive→write" state machine maps perfectly to our daily research workflow. Scout phase gets read/search tools only. Deep-dive gets full access. Write phase gets file-write but no shell.
- **Sub-machine for verification**: The "write" state could invoke a "fact-check" sub-machine before transitioning to "done."
- **Observation**: Statewright's MCP-based enforcement wouldn't work directly for agent-infra's Python-based runner, but the engine is embeddable (Rust crate, `use statewright_engine`). We could either embed it via PyO3 or reimplement the state machine pattern in Python — the logic is straightforward.

## 6. Follow-up Questions

1. **Full SWE-bench**: What happens when you run Statewright-constrained agents on the full 2,294-instance SWE-bench? Does the 2/10→10/10 ratio hold at scale, or does it regress?
2. **State machine generation quality**: How good are agent-generated state machines? What's the failure mode — over-constrained (agent gets stuck) or under-constrained (agent runs wild)?
3. **Composability**: Can you compose state machines without deadlocks? The `Invoke` pattern is promising but what happens when sub-machine A invokes sub-machine B which invokes sub-machine A?
4. **firn integration feasibility**: How hard would it be to add protocol-level tool gating to firn's ToolExecutor? firn already has a `ToolRegistry` — adding `allowed_tools: Optional[set[str]]` per phase is straightforward, but the hard part is propagating phase context through the call chain.
5. **Learning workflows**: Could we learn optimal tool constraints from agent traces? If we log which tools the agent actually needed vs. which it had access to, we could auto-tighten constraints.
6. **Comparison to A2A/MCP-native guardrails**: Google's Agent-to-Agent protocol and Anthropic's MCP both have security primitives. How does Statewright's approach compare? Is there a convergence toward standard agent guardrail protocols?
