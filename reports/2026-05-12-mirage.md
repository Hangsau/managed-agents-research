## 1. The Problem

AI agent 要操作各種後端服務（S3、Slack、GitHub、Gmail、Redis），每個服務都有自己的 SDK、認證方式、API 語義。Agent 要學 N 套接口，開發者要維護 N 個 action adapter。

## 2. Core Mechanism

Mirage 把所有後端掛載成一個**統一虛擬檔案系統（VFS）**。Agent 只需要會 bash 就能操作所有服務：

```python
ws = Workspace({
    '/data':  RAMResource(),
    '/s3':    S3Resource(...),
    '/slack': SlackResource(...),
})
await ws.execute('grep alert /slack/general/*.json | wc -l')
await ws.execute('cp /s3/report.csv /data/local.csv')
```

- 每個 resource 實作相同的檔案系統語義（read, write, list, stat）
- Agent 用 `cat`, `grep`, `cp`, `wc`, `jq` 等標準工具操作
- Pipeline 可以跨服務組合（`|` pipe 支援）

## 3. Why It Matters

- **零新詞彙成本**：LLM 已經在大量 bash/unix 語料上訓練過
- **單一抽象**：N 個 SDK → 1 個檔案系統
- **可移植**：workspace 可以 snapshot、clone、版本控制
- **多語言**：Python + TypeScript SDK，支援 OpenAI Agents SDK、LangChain、Pydantic AI

## 4. Limitations / Honest Assessment

- **Alpha 階段**：v0.0.1-alpha.1，API 可能大改
- **平台限制**：FUSE-based mounts 需要 macOS/Linux
- **安全風險**：agent 有權限操作所有掛載資源，沒有內建的 semantic verification（參見 Recursive Deletion 事件）
- **資源覆蓋率**：雖然支援很多服務，但每個 resource 的實作品質不一

## 5. Actionable for Our Projects

### managed-agents
目前 action_executor 每個 action 是獨立的（web_search, bash, file_read...）。可以考慮引入「統一操作語義」層：
- 外部服務（S3, Slack, GitHub）不寫獨立 action，而是掛載到 VFS
- playbook step 的 `action: "bash"` 搭配 Mirage-style workspace 就能操作任何服務
- 減少 action 數量，降低 agent 的學習負擔

### firn
如果 firn 需要操作外部服務（讀取 Gmail、寫 Slack、查 GitHub issue），Mirage 的設計可以直接嵌入 skill 裡：
```python
from mirage import Workspace
ws = Workspace({"/github": GitHubResource(...)})
await ws.execute("cat /github/firn/issues/open.json")
```

```bash
uv venv mirage-test

uv pip install mirage-ai
```

基本 bash execution 可運作，回傳 IOResult（stdout/stderr/exit_code）。RAMResource 預設 read-only，需額外配置才能寫入。
