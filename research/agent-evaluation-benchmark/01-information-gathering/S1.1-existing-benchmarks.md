# S1.1: Existing Agent Benchmarks — Information Gathering

Research cutoff: 2026-05-15. This document surveys 13 existing benchmarks for agentic systems, focusing on their design, evaluation approach, and coverage dimensions.

---

### SWE-bench / SWE-bench Verified

- **Source URL(s)**: [SWE-bench Leaderboards](https://www.swebench.com/), [GitHub](https://github.com/SWE-bench/SWE-bench), [OpenAI announcement](https://openai.com/index/introducing-swe-bench-verified/)
- **Covered dimensions**: Code generation, software engineering, repository-level reasoning, issue resolution, version control integration
- **Task form**: Agentic (multi-turn interaction with Git, file system, test runner)
- **Scoring method**: Unit-test pass/fail, behavioral-check (patch correctness via test suite)
- **Framework-agnostic score (1-5)**: 3 — Requires repository checkout, Git CLI, Python/shell execution, and test infrastructure. Can be plugged into different agent scaffolds (e.g., SWE-Agent) but tied to Unix-like execution environment and repo format.
- **Strengths**: 
  - Real GitHub issues (2,294 in full set) reflect production software engineering tasks
  - Executable evaluation via unit tests provides objective ground truth
  - SWE-bench Verified (500 samples) adds human validation to curate high-quality instances
- **Weaknesses / limitations**: 
  - Limited to Python repositories; not multilingual
  - Requires local environment setup (Git, test runners, package managers)
  - Highest-performing frontier models score <25% on SWE-bench Pro, indicating ceiling effects diminish signal
- **Relevance to deployed-system framing**: Directly applicable; treats agent as a black box that receives repository context and must interact with standard development tools. Score reflects end-to-end capability.

### SWE-bench Live

- **Source URL(s)**: [SWE-bench Leaderboards](https://www.swebench.com/)
- **Covered dimensions**: Real-time software engineering, dynamic issue tracking, repository updates
- **Task form**: Agentic (interaction with live GitHub APIs)
- **Scoring method**: Behavioral-check (verified PR merge success)
- **Framework-agnostic score (1-5)**: 2 — Tightly coupled to GitHub API authentication, rate limits, and live repository state. Different agents will see different repository snapshots, limiting reproducibility.
- **Strengths**: 
  - Reflects actual deployment conditions (dynamic repositories)
  - Prevents benchmark saturation by continuously adding new issues
  - Directly measures real-world impact (merged vs. rejected PRs)
- **Weaknesses / limitations**: 
  - Non-reproducible across agents or time (repository state changes)
  - Requires GitHub authentication and rate-limit handling
  - Not suitable for offline evaluation or rapid iteration
- **Relevance to deployed-system framing**: Measures live operational capability but sacrifices reproducibility. Valuable for assessing agents in production but not suitable as a baseline benchmark for comparing system architectures.

### GAIA

- **Source URL(s)**: [HAL Leaderboard](https://hal.cs.princeton.edu/gaia), [ArXiv](https://arxiv.org/abs/2311.12983), [Meta Research](https://ai.meta.com/research/publications/gaia-a-benchmark-for-general-ai-assistants/)
- **Covered dimensions**: Multi-step reasoning, web search, tool use, RAG, knowledge synthesis, information retrieval
- **Task form**: Open-end (free-form goal specification with verification by answer matching)
- **Scoring method**: Exact-match (answer verification), LLM-judge (fuzzy match for numerical/categorical answers)
- **Framework-agnostic score (1-5)**: 4 — Minimal infrastructure dependency; agents receive text goal, return text answer. Supports diverse tool integrations (search APIs, code execution, etc.) without prescribing architecture.
- **Strengths**: 
  - 466 real-world questions reflect human assistant tasks (92% human accuracy baseline)
  - Simple I/O contract (question → answer) supports any agent architecture
  - Leaderboard infrastructure enables public comparison (300 held-out answers)
- **Weaknesses / limitations**: 
  - Answer evaluation often brittle (exact-match for numerical answers, fuzzy for text)
  - No execution environment provided; agents must integrate external APIs independently
  - Limited signal for agents scoring <15% (most models perform 15% vs. 92% human)
- **Relevance to deployed-system framing**: Frame-agnostic measure of goal-oriented reasoning; the black box need only emit final answer. Scales well across deployed architectures but lacks granular failure diagnosis.

### GAIA-2

- **Source URL(s)**: [ArXiv](https://arxiv.org/abs/2602.11964), [OpenReview](https://openreview.net/forum?id=9gw03JpKK4)
- **Covered dimensions**: Temporal reasoning, asynchronous environments, multi-agent coordination, dynamic state, ambiguity resolution, time-aware decision-making
- **Task form**: Interactive / agentic (agents interact with smartphone-like environment; environment evolves independently)
- **Scoring method**: Action-level verification, pass@1 (task completion)
- **Framework-agnostic score (1-5)**: 4 — Built on Agents Research Environments (ARE) platform; supports any agent that can issue structured actions. Action verifier is decoupled from agent architecture.
- **Strengths**: 
  - 1,120 human-annotated tasks in realistic smartphone environment
  - Action-level verification enables fine-grained feedback and RL training
  - Tests time-sensitive reasoning (frontier models struggle with latency constraints)
  - Published infrastructure (ARE framework) enables extensions
- **Weaknesses / limitations**: 
  - Asynchronous evaluation is non-deterministic across runs; hard to isolate agent capability from timing variance
  - No model yet dominates across all capability dimensions (GPT-5 @ 42%, Claude @ ~35%)
  - Smartphone domain narrows applicability vs. general-purpose agent tasks
- **Relevance to deployed-system framing**: Captures deployed-system realism (dynamic environments, temporal constraints); reveals trade-offs between speed and accuracy. Useful for assessing real-time deployment readiness.

### AgentBench

- **Source URL(s)**: [GitHub](https://github.com/THUDM/AgentBench), [ArXiv](https://arxiv.org/abs/2308.03688), [OpenReview](https://openreview.net/forum?id=zAdUB0aCTQ)
- **Covered dimensions**: Multi-turn reasoning, decision-making, long-horizon planning, operating system tasks, database queries, knowledge graphs, web shopping, web browsing
- **Task form**: Agentic (multi-turn open-ended; solving turns estimated 5–50)
- **Scoring method**: LLM-judge (evaluates intermediate states and final outcomes), task-completion rate
- **Framework-agnostic score (1-5)**: 3 — Provides 8 diverse environments (OS, DB, KG, games, etc.) but each requires environment-specific protocol. Agents must adapt to heterogeneous APIs.
- **Strengths**: 
  - Covers 8 qualitatively different domains, testing generalization
  - Evaluated 29 models (API + OSS); shows clear disparity between frontier and 70B models
  - Practical multi-turn tasks with realistic reasoning requirements
- **Weaknesses / limitations**: 
  - Environment-specific evaluation criteria (no unified scoring method)
  - Relies on LLM-judge, which introduces bias and inconsistency
  - Limited to synthetic environments (not real-world systems)
- **Relevance to deployed-system framing**: Measures breadth across domains but heterogeneous evaluation hampers direct system comparison. Useful for assessing agent generalization but not ideal as unified benchmark.

### WebArena

- **Source URL(s)**: [Website](https://webarena.dev/), [GitHub](https://github.com/web-arena-x/webarena), [ArXiv](https://arxiv.org/abs/2307.13854), [Paper PDF](https://webarena.dev/static/paper.pdf)
- **Covered dimensions**: Web navigation, long-horizon tasks, e-commerce, forum interaction, collaborative software, content management, DOM understanding, multi-step workflows
- **Task form**: Agentic (long-horizon; average 3.3 variations per template, 812 total tasks)
- **Scoring method**: Behavioral-check (exact-match on final state, substring matching on outputs)
- **Framework-agnostic score (1-5)**: 3 — Containerized environments (Docker) provide reproducibility but tight coupling to web DOM representation (HTML/text) and container orchestration. Requires Selenium/browser automation or structured API.
- **Strengths**: 
  - 812 realistic long-horizon tasks with measurable success criteria
  - Fully functional websites from 4 real-world domains
  - Human baseline (78.24%) provides calibration reference
- **Weaknesses / limitations**: 
  - Original evaluation brittle (substring matching); WebArena Verified corrected this with type-aware matching
  - Requires container environment; not suitable for agents without web driver access
  - Best human-level performance baseline (78%) vs. GPT-4 (14.41%) reveals wide capability gap
- **Relevance to deployed-system framing**: Measures web-agent capability in realistic domains. Dockerized environment is reproducible but introduces infrastructure coupling. High human baseline useful for calibration.

### WebArena Verified

- **Source URL(s)**: [GitHub](https://github.com/ServiceNow/webarena-verified), [OpenReview](https://openreview.net/forum?id=CSIo4D7xBG)
- **Covered dimensions**: Web navigation, DOM grounding, task planning, environment reliability
- **Task form**: Agentic (same long-horizon structure as WebArena)
- **Scoring method**: Behavioral-check (type-aware exact matching, improved from original substring-match)
- **Framework-agnostic score (1-5)**: 3 — Same framework coupling as WebArena but tightened evaluation rigor. Corrects measurement ambiguity; does not change agent interface requirements.
- **Strengths**: 
  - Stricter evaluation methodology (type-aware matching, manual audits of all 812 tasks)
  - Retains containerized reproducibility of original
  - Removes ambiguous task specifications
- **Weaknesses / limitations**: 
  - Still bound to Docker/browser-automation paradigm
  - Repair of original tasks may reduce coverage of edge cases
- **Relevance to deployed-system framing**: Improves measurement signal without changing system interface. Useful when comparing multiple agents on same benchmark, but measurement improvement alone does not address capability gaps.

### OSWorld

- **Source URL(s)**: [Website](https://os-world.github.io/), [GitHub](https://github.com/xlang-ai/OSWorld), [ArXiv](https://arxiv.org/abs/2404.07972), [NeurIPS 2024](https://neurips.cc/virtual/2024/poster/97468)
- **Covered dimensions**: Multimodal agents, OS-level interaction, desktop GUI automation, file I/O, multi-app workflows, cross-application data movement, visual grounding
- **Task form**: Agentic (open-ended; 369 tasks involving real apps on Ubuntu/Windows/macOS)
- **Scoring method**: Execution-based (custom per-task verification scripts), behavioral-check
- **Framework-agnostic score (1-5)**: 2 — Requires multimodal input (screenshot + optionally HTML), real OS environment (Linux/Windows/macOS VMs), and GUI automation. Tightly coupled to visual grounding and OS-specific APIs. Not portable across architectures.
- **Strengths**: 
  - First benchmark for open-domain computer tasks on real operating systems
  - 369 tasks span diverse apps (LibreOffice, VLC, Thunderbird, VS Code) and realistic workflows
  - Execution-based evaluation provides reliable ground truth
  - OSWorld-Verified improves infrastructure (AWS parallelization, 50x speedup)
- **Weaknesses / limitations**: 
  - Frontier models achieve only 12.24% vs. 72.36% human baseline, indicating early-stage capability measurement
  - Requires visual grounding; not suitable for text-only agents
  - Heavy infrastructure cost (VMs, screenshot rendering, environment setup)
  - GUI elements and app behavior vary across OS versions and configurations
- **Relevance to deployed-system framing**: Measures real-world desktop agent capability but requires entire OS stack. Frame-dependent evaluation (visual grounding) limits applicability to agents without vision. High human baseline provides reality check.

### τ-Bench (TAU-Bench)

- **Source URL(s)**: [Website](https://taubench.com/), [GitHub](https://github.com/sierra-research/tau-bench), [ArXiv](https://arxiv.org/abs/2406.12045)
- **Covered dimensions**: Agent-user interaction, API tool use, domain-specific policies, multi-turn dialogue, policy compliance, simulated users
- **Task form**: Interactive (agent interacts with simulated user and domain APIs; user behavior simulated by LM)
- **Scoring method**: Behavioral-check (database state verification at end of conversation), pass^k metric (reliability over multiple trials)
- **Framework-agnostic score (1-5)**: 4 — Simple text I/O interface (user message → agent response). No hard dependency on specific tool APIs; simulates them. Portable across deployed agents.
- **Strengths**: 
  - Measures realistic user-agent-API interaction with policy constraints
  - Domain-specific (retail, airline); can be extended to other domains
  - Efficient evaluation via database state comparison
  - Pass^k metric captures reliability across stochastic trials
- **Weaknesses / limitations**: 
  - Simulated users may not reflect real user behavior (edge cases, adversarial inputs)
  - Limited to two domains (retail, airline); generalization unclear
  - Even state-of-the-art (gpt-4o) solves only 35.2% on τ-airline, suggesting high difficulty but also possible task ambiguity
- **Relevance to deployed-system framing**: Models deployed user-agent interaction realistically. Text I/O contract is frame-agnostic; domain policies test real-world compliance constraints.

### ARC-AGI / ARC-AGI-2

- **Source URL(s)**: [Website](https://arcprize.org/arc-agi/1), [GitHub](https://github.com/fchollet/ARC-AGI), [ARC-AGI-2 ArXiv](https://arxiv.org/abs/2505.11831)
- **Covered dimensions**: Abstract reasoning, few-shot learning, pattern induction, visual reasoning, algorithmic problem-solving
- **Task form**: Closed-end (given 3–5 input-output grid pairs, infer transformation rule and produce output for test input)
- **Scoring method**: Exact-match (grid output must match reference exactly)
- **Framework-agnostic score (1-5)**: 5 — Pure input-output contract (grids as structured data). No execution environment, API, or infrastructure dependency. Any agent capable of grid transformation can be evaluated.
- **Strengths**: 
  - Measures core abstract reasoning and generalization from minimal examples
  - Remained unsolved by AI until very recent frontier models (o3-preview @ 75% on ARC-AGI-1)
  - ARC-AGI-2 raises difficulty for frontier models, providing continued signal
  - Framework-agnostic: input-output only, no scaffolding or tool integration required
- **Weaknesses / limitations**: 
  - Highly specialized domain (grid transformations); limited applicability to deployed agents with external tools
  - Agents must infer rules without explanations or intermediate feedback
  - ARC-AGI-1 asymptotic performance (o3-preview 75%) may indicate dataset saturation approaching
  - Human baseline not consistently reported
- **Relevance to deployed-system framing**: Pure reasoning benchmark with no infrastructure coupling. Measures generalization and abstract thinking but decoupled from real-world agent tasks. Useful for fundamental reasoning capability but not for deployed-system assessment.

### MLE-Bench

- **Source URL(s)**: [GitHub](https://github.com/openai/mle-bench), [OpenAI announcement](https://openai.com/index/mle-bench/), [ArXiv](https://arxiv.org/abs/2410.07095)
- **Covered dimensions**: Machine learning engineering, model training, hyperparameter tuning, data preparation, experiment execution, Kaggle competition tasks
- **Task form**: Agentic (agents participate in 75 Kaggle competitions; multi-turn interaction with dataset, training loop, submission API)
- **Scoring method**: Cost-aware (Kaggle competition leaderboard ranking, bronze/silver/gold medal thresholds), behavioral-check (submission score)
- **Framework-agnostic score (1-5)**: 3 — Requires Kaggle API authentication, computational resources (training jobs), and handling of dataset downloads. Agent must manage experimentation workflow. Partially portable but resource-dependent.
- **Strengths**: 
  - Real Kaggle competitions reflect authentic ML engineering tasks
  - Cost-aware scoring (medal thresholds) provides practical calibration
  - Includes human baselines from actual Kaggle leaderboards
  - MLE-bench Lite (22 competitions) enables faster evaluation
- **Weaknesses / limitations**: 
  - Requires Kaggle authentication and participation rules compliance
  - Heavy computational cost (training jobs on agents' infrastructure)
  - Only 16.9% of competitions achieve bronze medal with best setup (o1-preview + AIDE), indicating low absolute performance
  - Leaderboard dynamics change over time (other participants' submissions)
- **Relevance to deployed-system framing**: Measures end-to-end ML engineering capability with real cost-aware scoring. Infrastructure and auth requirements limit portability; evaluation reflects resource constraints of deployed system.

### Aider Polyglot / Aider-Edit Leaderboard

- **Source URL(s)**: [Aider Leaderboards](https://aider.chat/docs/leaderboards/), [Polyglot Leaderboard](https://llm-stats.com/benchmarks/aider-polyglot), [ArXiv](https://arxiv.org/abs/2503.03656)
- **Covered dimensions**: Code generation, code editing, multilingual coding (C++, Go, Java, JavaScript, Python, Rust), error recovery, iterative refinement
- **Task form**: Agentic (agent receives problem, attempts solution, receives test feedback, refines)
- **Scoring method**: Unit-test pass/fail (problems considered solved if code passes all test cases)
- **Framework-agnostic score (1-5)**: 4 — Simple contract (text problem → edited code files). No environment dependency beyond standard language toolchains (compilers, interpreters). Portable across agent architectures.
- **Strengths**: 
  - 225 challenging Exercism problems calibrated for wide model range (5–50% expected)
  - Multilingual coverage tests generalization across languages
  - Two-attempt format (with error feedback) reflects realistic code-agent interaction
  - Well-maintained leaderboard with active model comparison
- **Weaknesses / limitations**: 
  - Limited to code editing; does not measure full software engineering (no repository, version control, long-horizon planning)
  - Only two attempts; limited failure analysis
  - Exercism problems may not reflect production code tasks
- **Relevance to deployed-system framing**: Measures core code-generation and error-recovery capability. Minimal infrastructure requirement (standard toolchains). Good for isolating code capability from other agent dimensions.

### BrowseComp

- **Source URL(s)**: [OpenAI announcement](https://openai.com/index/browsecomp/), [ArXiv](https://arxiv.org/abs/2504.12516), [Paper PDF](https://cdn.openai.com/pdf/5e10f4ab-d6f7-442e-9508-59515c65e35d/browsecomp.pdf)
- **Covered dimensions**: Web browsing, information retrieval, persistence, multi-hop reasoning, factuality verification, complex search strategies
- **Task form**: Open-end (agent receives goal/question, returns answer; goal may require extended browsing)
- **Scoring method**: Exact-match / LLM-judge (answer correctness; human-rated for ambiguous answers)
- **Framework-agnostic score (1-5)**: 3 — Requires web browsing capability (search API, web scraper, or browser automation). Depends on external internet state; non-reproducible across time.
- **Strengths**: 
  - 1,266 challenging problems targeting hard-to-find, multi-hop facts
  - Tests realistic information-seeking behavior (persistence, strategy, reasoning)
  - Simple I/O contract (goal → answer)
  - High human baseline (92%) makes frontier model performance (1–2% with browsing enabled) interpretable
- **Weaknesses / limitations**: 
  - Non-reproducible; internet content and search results evolve
  - GPT-4o baseline (0.6% to 1.9% with browsing) indicates benchmark may be too hard or agents lack effective browsing strategy
  - No intermediate milestones or partial credit
  - Evaluator (human or LLM-judge) introduces subjective variation
- **Relevance to deployed-system framing**: Measures real-world information-seeking capability but at cost of reproducibility. Reflects deployed-agent challenge (dynamic external state). Useful for assessing production readiness but not for controlled system comparison.

### VisualWebArena

- **Source URL(s)**: [Website](https://jykoh.com/vwa), [GitHub](https://github.com/web-arena-x/visualwebarena), [ArXiv](https://arxiv.org/abs/2401.13649)
- **Covered dimensions**: Multimodal agents, visual grounding, web navigation, visual reasoning, image-text integration, GUI interaction
- **Task form**: Agentic (long-horizon visual web tasks; 910 tasks across Classifieds, Shopping, Reddit)
- **Scoring method**: Behavioral-check (task completion on real websites)
- **Framework-agnostic score (1-5)**: 2 — Requires multimodal input (screenshot + HTML + bounding-box annotations via Set-of-Marks). Tightly coupled to visual grounding and browser automation. Not portable to text-only agents.
- **Strengths**: 
  - Addresses gap in visual web-agent evaluation
  - 910 tasks provide broad coverage
  - Set-of-Marks (SoM) annotation standardizes visual grounding without large vision models
  - Human performance (78%) established as baseline
- **Weaknesses / limitations**: 
  - Text-only models perform poorly (GPT-4 @ 7.25%, GPT-4V @ 15.05%); large gap indicates benchmark difficulty may limit signal
  - Requires visual grounding; excludes text-only agents
  - Heavy infrastructure cost (screenshot rendering, DOM parsing, bounding-box annotation)
  - Reuses Shopping and Reddit from WebArena; limited to visual variants
- **Relevance to deployed-system framing**: Extends web evaluation to multimodal agents. Visual requirement limits scope but captures real-world agent interface (GUI as primary input). Large capability gap suggests benchmark is still discovering visual-agent frontier.

### InterCode

- **Source URL(s)**: [Website](https://intercode-benchmark.github.io/), [GitHub](https://github.com/princeton-nlp/intercode), [ArXiv](https://arxiv.org/abs/2306.14898)
- **Covered dimensions**: Interactive coding, code generation with execution feedback, shell commands, SQL queries, Python debugging, multi-language tool use
- **Task form**: Interactive / agentic (agent receives NL request, issues code commands, receives execution output; loop until success/failure)
- **Scoring method**: Task-completion (task considered solved when execution output indicates success)
- **Framework-agnostic score (1-5)**: 4 — Lightweight framework (Docker sandboxes for Bash/SQL/Python). Agent I/O contract is language-agnostic (code text → execution output text). Extensible to new environments.
- **Strengths**: 
  - Framework-agnostic design supports multiple code environments (Bash, SQL, Python)
  - Execution feedback enables iterative refinement
  - Docker-based safety and reproducibility
  - Compatible with standard prompting strategies (ReAct, Plan & Solve)
  - Extensible (e.g., InterCode-CTF for cybersecurity)
- **Weaknesses / limitations**: 
  - Limited to code-based interactions; does not cover GUI, API, or network tasks
  - No human baseline provided (difficulty calibration unclear)
  - Sparse evaluation results in literature; unclear frontier model performance
- **Relevance to deployed-system framing**: Lightweight, reproducible, environment-agnostic design. Useful for isolating code-based agent capability. Execution feedback enables real-time learning but requires safe sandbox environment.

### CyberSecEval / CyberSecEval 3 / CyberSecEval 4

- **Source URL(s)**: [CyberSecEval 3 ArXiv](https://arxiv.org/pdf/2408.01605), [CyberSecEval 4 GitHub](https://github.com/meta-llama/PurpleLlama/tree/main/CybersecurityBenchmarks), [Meta Documentation](https://meta-llama.github.io/PurpleLlama/CyberSecEval/docs/intro)
- **Covered dimensions**: Cybersecurity risks (offensive and defensive), vulnerability detection, automated social engineering, exploit development, vulnerability remediation, security policy compliance
- **Task form**: Agentic / interactive (agents interact with CTF challenges, vulnerability codebases, social engineering scenarios)
- **Scoring method**: Behavioral-check (successful exploit / vulnerability finding / remediation), LLM-judge (policy compliance)
- **Framework-agnostic score (1-5)**: 2 — CyberSecEval 3/4 focus on specific security domains (web vulnerabilities, CTF, malware). Requires specialized security tools (fuzzing, code analysis, network simulation). Not portable across general-purpose agents.
- **Strengths**: 
  - Covers both offensive and defensive security capabilities
  - Addresses real-world security concerns (social engineering, autonomous exploitation)
  - CyberSecEval 4 enables benchmarking defender tools
  - Evaluated across major LLM providers (OpenAI, Anthropic, Google, Meta)
- **Weaknesses / limitations**: 
  - Highly specialized domain; not suitable for general-purpose agent comparison
  - Offensive capability measurement raises ethical concerns (models should not exceed safety bounds)
  - CyberSecEval 4 requires specialized infrastructure (code analysis tools, sandboxed environments)
  - Limited public benchmarking data (many results remain proprietary)
- **Relevance to deployed-system framing**: Specialized evaluation for security-focused agents. Measures both capability and safety boundaries. Not suitable for general-purpose deployed-system assessment.

## Cross-cutting Observations

**Evaluation approach variation**: Benchmarks span three scoring paradigms — exact-match (ARC-AGI, MATH), unit-test / behavioral verification (SWE-bench, OSWorld, InterCode), and LLM-judge (AgentBench, GAIA). Exact-match is most reproducible but requires discrete outputs; behavioral checks are reliable but environment-dependent; LLM-judge is flexible but introduces evaluator variance.

**Framework coupling**: Benchmarks cluster into two groups: infrastructure-light (GAIA, ARC-AGI, τ-Bench use text I/O; score 4–5 on agnosticism) and infrastructure-heavy (OSWorld, VisualWebArena, MLE-Bench require OS/cloud/visual access; score 2–3). This coupling determines whether evaluation can transfer across deployed-system architectures.

**Frontier model saturation**: ARC-AGI-1 shows saturation signals (o3 @ 75%); WebArena and GAIA show wide variance (GPT-4 @ 14% vs. human 78%), suggesting either benchmark difficulty misalignment or genuine capability gaps. Most benchmarks do not yet exhibit ceiling effects at frontier-model performance.

**Coverage absence**: No benchmark yet evaluates pure reasoning over distributed state, negotiation, or adversarial interaction. Agent benchmarks focus on task completion, not on robustness to adversarial input or behavior under distribution shift.

**Reproducibility trade-off**: Benchmarks requiring live external systems (BrowseComp, SWE-bench Live, MLE-Bench) sacrifice reproducibility for realism. Static benchmarks (ARC-AGI, GAIA, WebArena) enable controlled comparison but may not reflect production agents' actual deployment challenges.

**Human baselines rare**: Only 5 benchmarks report human performance (GAIA @ 92%, WebArena @ 78%, OSWorld @ 72%, VisualWebArena @ 78%, BrowseComp @ 92%). This limits calibration of benchmark difficulty.
