# S1.8: Black-Box & Contract Testing Methodologies from Software Engineering

Investigation of black-box testing, contract testing, fuzzing, property-based testing, differential testing, chaos engineering, and end-to-end testing pyramid approaches for AI system capability assessment.

---

## Black-Box Testing Fundamentals

### Equivalence Partitioning
- **Source URL**: [GeeksforGeeks Software Testing](https://www.geeksforgeeks.org/software-testing/), [StickyMinds](https://www.stickyminds.com/article/using-equivalence-partitioning-and-boundary-value-analysis-black-box-testing)
- **Core idea**: Divide input domain into equivalence classes where all values are processed identically; test one representative from each partition to reduce test count while maintaining coverage.
- **What property of the system it verifies**: Correctness of data validation logic; ability to classify inputs into distinct behavioral categories.
- **Heterogeneity tolerance**: 5/5 — requires zero knowledge of internal classification mechanism; purely behavioral.
- **Relevance to AI system testing**: Can partition input prompts or queries into functional equivalence classes (e.g., knowledge-retrieval, reasoning, creative-generation) and verify each category produces consistent output properties.
- **Concrete artifacts**: Manual test design; frameworks like TestRail, Zephyr, Azure Test Plans support partition documentation.

### Boundary Value Analysis (BVA)
- **Source URL**: [Software Testing Help](https://www.softwaretestinghelp.com/what-is-boundary-value-analysis-and-equivalence-partitioning/), [GeeksforGeeks](https://www.geeksforgeeks.org/software-testing/software-testing-boundary-value-analysis-vs-equivalence-partitioning/)
- **Core idea**: Test at partition edges and corner cases where behavior typically changes; a one-unit deviation from a boundary often reveals bugs invisible in partition interiors.
- **What property of the system it verifies**: Correctness of edge-case handling; detection of off-by-one errors, numeric overflows, truncation artifacts.
- **Heterogeneity tolerance**: 5/5 — boundary identification is behaviorally observable without implementation knowledge.
- **Relevance to AI system testing**: Test prompt length limits, token boundaries, context window edges, numeric parameter extremes (temperature 0.0 vs 1.0, top-k=1 vs k=vocabulary_size).
- **Concrete artifacts**: Manual test case design; can be integrated into parameterized test frameworks (pytest parametrize, JUnit Theories).

## Contract Testing

### Pact Framework (Consumer-Driven Contracts)
- **Source URL**: [Pact Docs](https://docs.pact.io/), [PactFlow CDC Guide](https://pactflow.io/what-is-consumer-driven-contract-testing/), [Baeldung](https://www.baeldung.com/pact-junit-consumer-driven-contracts)
- **Core idea**: Contracts are generated during consumer tests and verified against provider implementations independently; ensures both sides will interoperate without full system integration test.
- **What property of the system it verifies**: Interface contract conformance (request/response format, fields, types); compatibility between decoupled services; prevents breaking changes.
- **Heterogeneity tolerance**: 4/5 — framework-agnostic about provider internals; requires contractual agreement on message shape.
- **Relevance to AI system testing**: Test that AI system outputs conform to declared contract (specific JSON schema, field presence/type, response structure) regardless of internal model or prompt engineering changes.
- **Concrete artifacts**: [Pact Foundation frameworks](https://docs.pact.io/) (JVM, .NET, JavaScript, Python, Go, Ruby); [PactFlow broker](https://pactflow.io/) for contract storage and verification.

## Fuzzing Methodologies

### Coverage-Guided Fuzzing (libFuzzer, AFL)
- **Source URL**: [LLVM libFuzzer Documentation](https://llvm.org/docs/LibFuzzer.html), [Google Fuzzing Docs](https://github.com/google/fuzzing/blob/master/docs/afl-based-fuzzers-overview.md), [AFL++ In Depth](https://aflplus.plus/docs/fuzzing_in_depth/)
- **Core idea**: Instrument target to track edge coverage; mutate seed corpus; feed mutations that reach new code paths back into corpus; iteratively expand coverage via feedback loop.
- **What property of the system it verifies**: Memory safety (crashes, buffer overflows, use-after-free); undefined behavior; robustness under malformed/adversarial input; latent assertion failures.
- **Heterogeneity tolerance**: 3/5 — requires compilation instrumentation or binary rewriting; assumes access to target or its interface; limited to deterministic targets.
- **Relevance to AI system testing**: Fuzz prompt generation with mutation operators (token insertion, deletion, replacement) to discover edge cases, prompt injection vectors, context confusion, or latent reasoning failures; track token-level coverage expansion.
- **Concrete artifacts**: [libFuzzer](https://llvm.org/docs/LibFuzzer.html) (LLVM-based, in-process), [AFL++](https://aflplus.plus/) (fork-based, evolutionary), [GTest](https://github.com/google/googletest) integration, ClusterFuzz for distributed fuzzing.

## Property-Based Testing

### QuickCheck / Hypothesis Framework
- **Source URL**: [Hypothesis Documentation](https://hypothesis.works/articles/what-is-property-based-testing/), [QuickCheck GitHub](https://github.com/BurntSushi/quickcheck), [Increment: In Praise of Property-Based Testing](https://increment.com/testing/in-praise-of-property-based-testing/)
- **Core idea**: Define properties (invariants) that must hold; framework generates many test cases automatically; discovers minimal failing example via shrinking when invariant violated.
- **What property of the system it verifies**: Logical invariants and post-conditions across diverse input spaces; commutativity, idempotence, composition laws; edge cases missed by hand-written tests.
- **Heterogeneity tolerance**: 5/5 — purely behavioral; framework generates data, system under test responses; no access to internals required.
- **Relevance to AI system testing**: Verify invariants such as "output tokenization round-trips back to input," "response length is bounded," "semantic similarity increases with related prompts," "output respects declared constraints."
- **Concrete artifacts**: [Hypothesis](https://hypothesis.readthedocs.io/) (Python, mature), [QuickCheck](https://hackage.haskell.org/package/QuickCheck) (Haskell), [jqwik](https://jqwik.net/) (Java), [proptest](https://docs.rs/proptest/) (Rust).

## Differential Testing

### Multi-Implementation Comparison
- **Source URL**: [Wikipedia: Differential Testing](https://en.wikipedia.org/wiki/Differential_testing), [ScienceDirect: Validating Heterogeneous Knowledge Bases](https://www.sciencedirect.com/science/article/abs/pii/0957417495000275), [Emergent Mind](https://www.emergentmind.com/topics/differential-testing)
- **Core idea**: Run identical inputs against two or more independent implementations; observe behavior divergence; discrepancies flag potential faults without requiring a complete oracle.
- **What property of the system it verifies**: Semantic correctness (logic bugs, interpretation mismatches); specification conformance across variants; portability and consistency.
- **Heterogeneity tolerance**: 5/5 — purely comparative; requires no oracle, only multiple implementations; treats each as black box.
- **Relevance to AI system testing**: Compare outputs from same AI model at different temperatures, different model versions, different providers (e.g., Claude vs GPT vs Gemini) to detect reasoning divergences, consistency failures, and provider-specific quirks.
- **Concrete artifacts**: NEZHA (domain-independent differential testing tool), manual side-by-side comparison frameworks, cross-provider API testing harnesses.

## Chaos Engineering

### Failure Injection (Chaos Monkey, Gremlin)
- **Source URL**: [Gremlin Chaos Monkey Guide](https://www.gremlin.com/chaos-monkey), [Netflix Chaos Monkey](https://netflix.github.io/chaosmonkey/), [Gremlin Chaos Engineering](https://www.gremlin.com/chaos-engineering)
- **Core idea**: Intentionally inject failures (instance termination, resource constraints, network degradation) to test resilience; observe recovery behavior, failover effectiveness, alert responsiveness.
- **What property of the system it verifies**: Resilience and fault tolerance; graceful degradation; recovery time objectives (RTO); operational readiness of incident response.
- **Heterogeneity tolerance**: 2/5 — requires infrastructure access and deployment capability; system-specific failure modes; not purely black-box (needs blast radius containment planning).
- **Relevance to AI system testing**: Inject prompt disruptions (truncation, token corruption, context loss), API latency spikes, rate-limiting scenarios; verify system behavior under constraint (e.g., token budget exhaustion, inference timeout).
- **Concrete artifacts**: [Gremlin](https://www.gremlin.com/) (SaaS platform, 400+ enterprises), [Chaos Monkey](https://netflix.github.io/chaosmonkey/) (open-source, AWS-focused), [Chaos Mesh](https://chaos-mesh.org/) (Kubernetes-native), [LitmusChaos](https://litmuschaos.io/) (CNCF).

## State-Transition & Decision-Table Testing

### State-Transition Testing
- **Source URL**: [GeeksforGeeks State Transition Testing](https://www.geeksforgeeks.org/software-engineering/state-transition-testing/), [Guru99](https://www.guru99.com/state-transition-testing.html), [TestSigma](https://testsigma.com/blog/state-transition-testing/)
- **Core idea**: Model system as finite state machine; test transitions between states under specific inputs; verify correct next-state reached and output produced.
- **What property of the system it verifies**: Behavioral correctness of stateful systems; state machine invariants; valid transition sequences; prevention of invalid state combinations.
- **Heterogeneity tolerance**: 4/5 — requires state-machine abstraction (behavioral model), not internal implementation; can be reverse-engineered from I/O traces.
- **Relevance to AI system testing**: Model conversation state progression (initial → responding → clarifying → concluding); verify state transitions respect conversation history constraints and output consistency per state.
- **Concrete artifacts**: UML state diagrams, state transition tables, tools like GraphWalker, UPPAAL for formal verification.

### Decision Table Testing
- **Source URL**: [Tutorials Hut Decision Table Testing](https://tutorialshut.com/black-box-testing-techniques/decision-table-state-transition-testing/), [Ice House Indonesia](https://icehousecorp.com/decision-table-testing-state-transition-error-guessing/)
- **Core idea**: Construct table mapping input combinations to expected outputs; each row is a test case; systematic coverage of complex business logic branches.
- **What property of the system it verifies**: Correctness of complex conditional logic; completeness of rule coverage; absence of contradictory rules.
- **Heterogeneity tolerance**: 5/5 — purely behavioral; no implementation knowledge required.
- **Relevance to AI system testing**: Model AI output rules (e.g., "if prompt mentions financial advice AND user not verified → refuse"; "if token count > threshold → summarize input"); verify all rule combinations produce correct response category.
- **Concrete artifacts**: Spreadsheet-based tables, automated decision table tools (e.g., TestRail, AgileTest).

## End-to-End Testing Pyramid

### Test Automation Pyramid (Unit / Integration / E2E)
- **Source URL**: [Martin Fowler Practical Test Pyramid](https://martinfowler.com/articles/practical-test-pyramid.html), [CircleCI Testing Pyramid](https://circleci.com/blog/testing-pyramid/), [Home Office Engineering Standards](https://engineering.homeoffice.gov.uk/standards/test-pyramid/)
- **Core idea**: Inverse-pyramid distribution — many fast unit tests (isolated component verification), fewer integration tests (component interactions), minimal E2E tests (full workflow simulation); reduces feedback latency and maintenance cost.
- **What property of the system it verifies**: Layered correctness from isolated units → compositions → end-user workflows; failure isolation (unit failure vs integration vs orchestration).
- **Heterogeneity tolerance**: Varies by layer: Unit (5/5 black-box), Integration (4/5 interface-dependent), E2E (2/5 requires full deployment).
- **Relevance to AI system testing**: Unit level: verify prompt formatting, tokenization, response parsing. Integration: test prompt→LLM→output pipeline per component. E2E: verify multi-turn conversation, tool usage, fallback behavior in deployed system.
- **Concrete artifacts**: pytest (Python), JUnit (Java), RSpec (Ruby), Cypress/Playwright (E2E), Docker compose for test environments.

## Concepts Mapping Table

| SE Technique | AI Agent Evaluation Analog |
|---|---|
| Equivalence Partitioning | Input prompt category clustering (knowledge-retrieval, reasoning, creative tasks); verify consistent behavior per category |
| Boundary Value Analysis | Test prompt length limits, token boundaries, context window edges, parameter extremes (temperature, top-k) |
| Pact Contract Testing | Verify AI output conforms to declared schema/structure across model version/temperature changes |
| Coverage-Guided Fuzzing | Mutate prompts to discover edge cases, prompt injection vectors, reasoning failures; track semantic coverage expansion |
| Property-Based Testing | Verify logical invariants (output bound, tokenization reversibility, constraint satisfaction) across generated prompt space |
| Differential Testing | Compare outputs across model versions, providers, temperatures to detect semantic divergences |
| Chaos Engineering | Inject prompt truncation, API latency, rate-limiting; verify graceful degradation and recovery behavior |
| State-Transition Testing | Model conversation state progression; verify transitions respect history constraints and output consistency |
| Decision-Table Testing | Systematically test output rule combinations (refuse-cases, tone-matching, constraint-enforcement) |
| E2E Pyramid | Layered testing: unit (component isolation) → integration (pipeline) → end-to-end (deployed system behavior) |

## Transferable Principles

**Coverage as feedback mechanism**: Fuzzing's core insight—that coverage expansion is a computable signal guiding search—transfers directly: define "semantic coverage" metrics for AI outputs (reasoning depth, constraint satisfaction, edge-case handling) and use them to prioritize test case generation, similar to how libFuzzer prioritizes inputs that reach new code paths.

**Contract-first design**: Pact's consumer-driven contract principle (define integration expectations first, verify independently) maps to AI systems: declare expected output properties (JSON schema, logical invariants, safety constraints) upfront, then verify compliance across model changes without full integration testing.

**Stateless black-box verification**: Differential testing and property-based testing require zero implementation knowledge; they treat systems as pure input→output functions. This principle is critical for AI: many deployed models are closed-source; verification must remain behavioral and provider-agnostic.

**Layered resilience testing**: Chaos engineering's graduated failure injection (from single-instance outages to cascading faults) mirrors AI robustness testing: start with isolated prompt stress (boundary values), escalate to interaction patterns (multi-turn, tool chains), end with full-system constraints (concurrency, resource depletion).

**Equivalence abstraction**: Both equivalence partitioning and state-transition testing succeed because they abstract away implementation details and focus on observable behavior categories. For AI, this means defining prompt equivalence classes *before* scrutinizing model internals.
