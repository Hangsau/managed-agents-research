# S1.4 — Built-in Evaluation Tooling by Agent Frameworks

## LangChain / LangSmith — LangSmith Evaluation Platform

- **Source URL**: [LangSmith Evaluation](https://docs.langchain.com/langsmith/evaluation), [LangChain Evaluators Reference](https://reference.langchain.com/python/langsmith/evaluation)
- **What dimensions does it measure**: Correctness, quality of generated responses, heuristic checks, LLM-as-judge scoring, pairwise comparisons, safety metrics, trajectory evaluation, user behavior patterns, multimodal output quality
- **How it interfaces with the framework**: Python SDK (`langsmith.evaluation`), web UI dashboard, pytest/Vitest/GitHub Actions CI/CD plugins, REST API, native LangChain integration
- **Can it test a non-native system (3rd-party agent)?**: Yes — via REST API and generic evaluator templates; requires adapter code to serialize inputs/outputs
- **Scoring approach**: Multi-evaluator: human annotation queues, heuristic rules, LLM-as-judge with configurable criteria, pairwise ranking, regression detection against baseline metrics
- **Framework-agnostic score (1-5)**: 3 — Native to LangChain but evaluators (30+ templates) are reusable; API interface allows wrapping external systems with overhead
- **Notable strengths/weaknesses**: 
  - Strengths: Production monitoring in real-time; 30+ pre-built templates; integrates into CI/CD pipelines natively
  - Weaknesses: Tight coupling with LangSmith cloud dashboard; evaluating external agents requires manual endpoint wrapping

## LlamaIndex — Eval Module

- **Source URL**: [LlamaIndex Evaluating Guide](https://developers.llamaindex.ai/python/framework/module_guides/evaluating/), [Evaluation Examples](https://docs.llamaindex.ai/en/stable/understanding/evaluating/evaluating/)
- **What dimensions does it measure**: Correctness (vs. reference answers), semantic similarity, faithfulness (hallucination detection), context relevancy, answer relevancy, guideline adherence
- **How it interfaces with the framework**: Python module (`llama_index.evaluation`), integrates with RAG pipelines, evaluators applied post-retrieval and post-generation stages
- **Can it test a non-native system (3rd-party agent)?**: Partial — can wrap external systems but designed for LlamaIndex components; requires custom adapter for non-LlamaIndex LLMs
- **Scoring approach**: LLM-based evaluation using prompt instructions, comparison against ground-truth labels (optional), ranking metrics for retrieval (MRR, hit-rate, precision)
- **Framework-agnostic score (1-5)**: 2 — Tightly coupled to LlamaIndex data structures (Node, Document, Response); limited portable use outside ecosystem
- **Notable strengths/weaknesses**:
  - Strengths: Purpose-built for RAG; label-free evaluation option; modular evaluator design
  - Weaknesses: Requires LlamaIndex response objects; community integrations (Ragas, DeepEval) needed for broader coverage

## AutoGen — AgentEval & AutoGenBench

- **Source URL**: [AgentEval Framework](https://microsoft.github.io/autogen/0.2/blog/2024/06/21/AgentEval/), [AutoGenBench](https://microsoft.github.io/autogen/0.2/blog/2024/01/25/AutoGenBench/)
- **What dimensions does it measure**: Task utility (multi-dimensional), problem-solving correctness, agent trajectory validity, intermediate step quality, execution success rates
- **How it interfaces with the framework**: AgentEval embedded in AutoGen library (Python), AutoGenBench as standalone CLI tool (PyPI), Docker isolation for benchmarking
- **Can it test a non-native system (3rd-party agent)?**: Partial — AutoGenBench can run external agents with Docker adapters; AgentEval requires multi-agent scaffold integration
- **Scoring approach**: Three-agent ensemble (CriticAgent, QuantifierAgent, VerifierAgent) for multi-faceted scoring; dataset-based benchmark regression testing
- **Framework-agnostic score (1-5)**: 2 — AgentEval depends on AutoGen messaging protocol; AutoGenBench more flexible but Docker-bound
- **Notable strengths/weaknesses**:
  - Strengths: Multi-dimensional utility assessment; Docker isolation prevents test pollution
  - Weaknesses: AutoGen now in maintenance mode; CriticAgent approach adds latency; limited generalization beyond AutoGen

## CrewAI — Testing Module

- **Source URL**: [CrewAI Testing](https://docs.crewai.com/en/concepts/testing), [CrewAI GitHub](https://github.com/crewAIInc/crewAI)
- **What dimensions does it measure**: Crew output correctness, iteration stability, performance metrics (latency, token usage), agent role adherence
- **How it interfaces with the framework**: CLI command (`crewai test`), native integration with Crew composition, Python API hooks
- **Can it test a non-native system (3rd-party agent)?**: No — CLI designed exclusively for CrewAI-defined agents; external agents require third-party tool (Promptfoo, DeepEval)
- **Scoring approach**: Iteration-based sampling (configurable runs), deterministic output comparison, extensible via external frameworks
- **Framework-agnostic score (1-5)**: 1 — CrewAI testing is proprietary; external evaluation requires wrapping with Promptfoo, DeepEval, or ArkSim
- **Notable strengths/weaknesses**:
  - Strengths: Seamless crew integration; low-friction iteration testing
  - Weaknesses: Single-framework scope; weak evaluation semantics without external integrations

## Pydantic AI — Evals Module

- **Source URL**: [Pydantic Evals](https://pydantic.dev/docs/ai/evals/evals/), [Pydantic AI Testing](https://ai.pydantic.dev/testing/)
- **What dimensions does it measure**: Output correctness, execution trace quality, format compliance, tool call validity, behavioral assertions (reasoning paths, agent decisions)
- **How it interfaces with the framework**: Python-first API (code-based definition), datasets with cases and evaluators, integrates with Logfire observability
- **Scoring approach**: Custom evaluators (deterministic + LLM-based), span-level analysis of execution traces, built-in type validators, experiment reporting
- **Framework-agnostic score (1-5)**: 3 — Evals are Python functions; can wrap non-Pydantic agents but traces require agent instrumentation for full power
- **Notable strengths/weaknesses**:
  - Strengths: Code-first, flexible design; trace-level debugging with Pydantic models; Logfire integration for visualization
  - Weaknesses: Tracing power degrades outside Pydantic ecosystem; requires Logfire subscription for full observability

## DSPy — Evaluation & Metrics

- **Source URL**: [DSPy Metrics](https://dspy.ai/learn/evaluation/metrics/), [DSPy Evaluation Overview](https://dspy.ai/learn/evaluation/overview/)
- **What dimensions does it measure**: Exact match, semantic F1 (recall + precision), grounded completeness, custom metric definitions via DSPy programs
- **How it interfaces with the framework**: Python module, metrics are user-defined functions or DSPy Signatures, integrated with Evaluate class for parallel execution
- **Can it test a non-native system (3rd-party agent)?**: Yes — metrics are functions; can evaluate any system's output given input/output serialization
- **Scoring approach**: User-defined metrics (simple functions or DSPy programs), Evaluate class runs metrics in parallel, supports metric optimization/compilation
- **Framework-agnostic score (1-5)**: 4 — Metrics are composable, model-agnostic functions; can chain with any system producing text outputs
- **Notable strengths/weaknesses**:
  - Strengths: Metric optimization (compile your metric); highly composable; non-opinionated on what to measure
  - Weaknesses: Minimal built-in metrics; users must implement domain-specific scorers; no pre-built benchmark datasets

## Haystack — Evaluators Module

- **Source URL**: [Haystack Evaluation](https://docs.haystack.deepset.ai/docs/evaluation), [Evaluators Reference](https://docs.haystack.deepset.ai/reference/evaluators-api)
- **What dimensions does it measure**: Document relevance, answer faithfulness, context relevance, answer exactness, semantic answer similarity, custom aspects via LLMEvaluator
- **How it interfaces with the framework**: Python components (Pipeline-native), evaluators as composable nodes, integrates with Haystack pipeline DAGs
- **Can it test a non-native system (3rd-party agent)?**: Partial — evaluators (LLMEvaluator, RagasEvaluator) can score external outputs but lose pipeline integration
- **Scoring approach**: Model-based (LLM grading) and statistical (MRR, recall, precision); label-free operation via prompt instructions; few-shot label enhancement
- **Framework-agnostic score (1-5)**: 3 — Evaluators are composable but assume Haystack Document/Answer types; external integration requires serialization
- **Notable strengths/weaknesses**:
  - Strengths: Label-free evaluation; statistical and LLM-based approaches coexist; Ragas integration for RAGAS metrics
  - Weaknesses: Type dependencies on Haystack objects; limited support for agent-specific metrics (tool calls, reasoning)

## Promptfoo — Evaluation & Red Teaming Framework

- **Source URL**: [Promptfoo Docs](https://www.promptfoo.dev/docs/intro/), [Promptfoo GitHub](https://github.com/promptfoo/promptfoo)
- **What dimensions does it measure**: Prompt correctness, output safety, jailbreak resistance, factuality, JSON schema compliance, RAG hallucination, coding correctness, user satisfaction
- **How it interfaces with the framework**: CLI-first (`promptfoo eval`), web UI, config-based (YAML) test definitions, language-agnostic (Python/JS/any)
- **Can it test a non-native system (3rd-party agent)?**: Yes — designed as system-agnostic; wraps any endpoint compatible with Chat Completions API or custom HTTP
- **Scoring approach**: Model-graded rubrics (G-Eval), regex/heuristic checks, expected output matching, custom JS/Python functions, benchmark datasets (100+ test cases)
- **Framework-agnostic score (1-5)**: 5 — Completely framework-agnostic; evaluates OpenAI, Claude, Gemini, Llama, and non-LLM systems via function calls
- **Notable strengths/weaknesses**:
  - Strengths: System-agnostic design; live reload for iteration; red-team vulnerability scanning; CI/CD integration
  - Weaknesses: Config-heavy for complex eval scenarios; limited built-in agent-specific metrics (tool use, reasoning chains)

## OpenAI Evals — Framework

- **Source URL**: [OpenAI Evals GitHub](https://github.com/openai/evals), [OpenAI Evals Guide](https://developers.openai.com/api/docs/guides/evals), [Agent Evals](https://developers.openai.com/api/docs/guides/agent-evals)
- **What dimensions does it measure**: Accuracy, safety, truthfulness, moderation, code correctness, agent behavior consistency, chat quality
- **How it interfaces with the framework**: Python library, JSON/YAML eval definitions, CLI runner, open-source registry of benchmarks, OpenAI API integration
- **Can it test a non-native system (3rd-party agent)?**: Yes — custom model endpoints support any Chat Completions API-compatible system
- **Scoring approach**: Exact match, fuzzy matching, LLM-as-judge (configurable model), custom Python graders, benchmark datasets
- **Framework-agnostic score (1-5)**: 4 — API-agnostic via Custom Completions endpoints; requires OpenAI-compatible interface
- **Notable strengths/weaknesses**:
  - Strengths: Open-source with registry; agent-specific evals; easy third-party model support
  - Weaknesses: OpenAI account required for scoring; eval definitions less concise than Promptfoo YAML; limited observability

## Anthropic — Evals SDK & Evaluation Tool

- **Source URL**: [Anthropic Evals Engineering Blog](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents), [Evaluation Tool Docs](https://docs.anthropic.com/en/docs/test-and-evaluate/eval-tool), [Anthropic Evals GitHub](https://github.com/anthropics/evals)
- **What dimensions does it measure**: Task correctness, agent trajectory validity, tool use appropriateness, multi-turn consistency, safety assertions
- **How it interfaces with the framework**: Evaluation tool embedded in Claude API dashboard, Python SDK (`anthropic-evals`), side-by-side model comparison UI
- **Can it test a non-native system (3rd-party agent)?**: Partial — Evaluation Tool is Claude-focused; Python SDK can wrap external agents but dashboard is Claude-only
- **Scoring approach**: 5-point quality grading, custom grader logic, task outcome validation, pairwise comparison with model variants
- **Framework-agnostic score (1-5)**: 2 — Evaluation Tool is Claude-exclusive; Python SDK more portable but lacks built-in evaluators
- **Notable strengths/weaknesses**:
  - Strengths: Lightweight side-by-side UI; seamless Claude API integration; prompt versioning for iteration
  - Weaknesses: Limited pre-built evaluators; Python SDK undocumented; agent-specific eval patterns require custom implementation

## TruLens — Evaluation & Tracing Framework

- **Source URL**: [TruLens Docs](https://www.trulens.org/), [TruLens GitHub](https://github.com/truera/trulens)
- **What dimensions does it measure**: Context relevance, groundedness (faithfulness), answer relevance (RAG triad), custom feedback metrics, LLM app execution traces
- **How it interfaces with the framework**: Python library, feedback function API, integrates with LangChain, LlamaIndex, and generic LLM apps via wrapper; OpenTelemetry (OTel) support
- **Can it test a non-native system (3rd-party agent)?**: Yes — feedback functions are model-agnostic; can wrap any system with execution tracing
- **Scoring approach**: Feedback functions (LLM-based + heuristic), RAG triad metrics, custom Metric class (new API), distributed evaluation with caching
- **Framework-agnostic score (1-5)**: 4 — Feedback functions decouple from framework; OTel support enables broad observability ecosystem integration
- **Notable strengths/weaknesses**:
  - Strengths: RAG triad well-researched; OTel interoperability; distributed eval with caching for scalability
  - Weaknesses: Learning curve for custom feedback functions; Metric API transition from Feedback (migration needed); requires instrumentation

## Inspect-AI (AISI) — Evaluation Framework

- **Source URL**: [Inspect-AI](https://inspect.aisi.org.uk/), [Inspect GitHub](https://github.com/UKGovernmentBEIS/inspect_ai), [Inspect Sandboxing](https://www.aisi.gov.uk/blog/the-inspect-sandboxing-toolkit-scalable-and-secure-ai-agent-evaluations)
- **What dimensions does it measure**: Task correctness, agent trajectory (multi-turn), tool use validity, reasoning quality, code execution correctness, security/jailbreak resistance
- **How it interfaces with the framework**: Python library (`inspect-ai`), Dataset → Task → Solver → Scorer opinionated primitives, sandboxed execution (Docker/Kubernetes/Proxmox)
- **Can it test a non-native system (3rd-party agent)?**: Yes — external solver support for Claude Code, Codex CLI, Gemini CLI, and custom agents via flexible interface
- **Scoring approach**: Text-based comparison, model-graded metrics, custom Python scorers, aggregated metrics from 200+ pre-built evals
- **Framework-agnostic score (1-5)**: 5 — Agent-agnostic; supports OpenAI, Anthropic, Google, Groq, Mistral, xAI, AWS Bedrock, Azure, local vLLM/Ollama
- **Notable strengths/weaknesses**:
  - Strengths: Sandboxed eval execution; 200+ pre-built evals; multi-provider model support; VS Code log viewer; Inspect View web UI
  - Weaknesses: Opinionated task structure requires adaptation; ecosystem still maturing; deployment on Proxmox/K8s less common

## Cross-framework patterns

### Shared dimensions across frameworks

All frameworks measure **task correctness** (exact match, semantic similarity, custom rubrics) and **safety/quality** (hallucination, faithfulness, adherence to guidelines). Most support both **label-free** (prompt-guided) and **label-based** (reference answer) evaluation. LangSmith, OpenAI Evals, and Promptfoo emphasize **monitoring in production**, while DSPy, Pydantic AI, and Inspect-AI focus on **development-time iteration**.

### Evaluation interface patterns

Four interface styles dominate: (1) **Python code-first** (DSPy, Pydantic, TruLens, Inspect-AI) — users define metrics/scorers programmatically; (2) **Config-based** (Promptfoo, OpenAI Evals) — YAML/JSON test definitions; (3) **Dashboard-native** (LangSmith, Anthropic) — web UI with limited code hooks; (4) **Pipeline-integrated** (LlamaIndex, Haystack) — evaluators as framework components. Promptfoo and Inspect-AI are system-agnostic (score 5); most others score 2–3 due to framework coupling or API dependencies.

### Scoring philosophy differences

LangSmith, LlamaIndex, and Haystack default to **LLM-as-judge** scoring; Promptfoo and OpenAI Evals support **multi-method** (fuzzy match + model grading + custom rules); Pydantic and DSPy enable **programmatic composition** of custom metrics; Inspect-AI **separates concerns** (Dataset, Solver, Scorer as orthogonal primitives). No framework presently offers **distributed federated evaluation** except TruLens (caching + scaling), making large-scale benchmark runs difficult across all except cloud-hosted solutions (LangSmith, Promptfoo).
