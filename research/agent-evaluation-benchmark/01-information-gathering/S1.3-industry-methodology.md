# S1.3: Industry Evaluation Methodology for Deployed AI Systems

## Anthropic — Claude SWE-bench Verified and System Cards (2025–2026)

### Anthropic — Claude Opus 4.7 SWE-bench Performance (2026)
- **Source URL**: https://www.anthropic.com/research/swe-bench-sonnet
- **Eval setup**: SWE-bench Verified: 500 human-reviewed GitHub issues from real Python repositories. Evaluated via flexible agentic scaffold with Bash tool and Edit Tool (string-replacement semantics, not line-numbers). Averaged over 25 trials with adaptive thinking, max effort, default sampling. Tasks range from 100k+ tokens across hundreds of turns per solution.
- **Public numbers reported**: Claude Opus 4.7 achieves 87.6% on SWE-bench Verified (up from 80.8% in prior release). Claude Sonnet 4 achieves 72.7% on public SWE-bench.
- **Methodology choices worth flagging**: 
  - Agent design emphasizes minimal scaffolding; language model given maximum control over action selection
  - Edit Tool uses old_str/new_str matching rather than line-numbers to prevent fragile modifications
  - Tool descriptions iteratively refined during development to prevent model misunderstandings
- **Limitations they acknowledge**: High token costs (100k+ per resolution), environment setup complexity, multimodal limitations for visual debugging
- **Framework-agnostic transferability (3/5)**: Setup is portable to third-party systems (any agentic framework can apply Bash+Edit scaffolding), but SWE-bench is repository-specific. Cognition and OpenAI have adapted SWE-bench for their own agents, showing moderate portability.

### Anthropic — Bloom: Automated Behavioral Evaluations (2025)
- **Source URL**: https://www.anthropic.com/research/bloom; https://alignment.anthropic.com/2025/bloom-auto-evals/
- **Eval setup**: Four-stage automated pipeline to measure behavioral properties (delusional sycophancy, self-preservation, instructed sabotage, etc.). Stages: Understanding → Ideation (scenario generation) → Rollout (execute scenarios in parallel) → Judgment (LLM-as-judge scoring on numerical scale + meta-judge suite report).
- **Public numbers reported**: Tested on 16 frontier models. Claude Opus 4.1 shows strongest correlation with human judgment (Spearman ρ=0.86). Successfully distinguishes intentionally misaligned "model organisms" from baseline in 9 of 10 test cases.
- **Methodology choices worth flagging**:
  - Researcher specifies behavior seed; agent generates detailed scenarios automatically to prevent manual evaluation bottleneck
  - Generates new scenarios on each run (prevents contamination) while measuring consistent underlying behaviors
  - Meta-judge produces suite-level report highlighting patterns across transcripts
- **Limitations they acknowledge**: Reliance on LLM judge quality; scenario generation may not cover edge cases outside training distribution; accessibility requires configuration expertise
- **Framework-agnostic transferability (4/5)**: Framework is designed to be model-agnostic and configurable. Any deployed system can be dropped into Bloom's rollout stage; behavior specification is system-independent.

### Anthropic — Claude Opus 4.5 / 4.6 Computer Use and OSWorld (2025)
- **Source URL**: https://assets.anthropic.com/m/64823ba7485345a7/Claude-Opus-4-5-System-Card.pdf (System Card); https://www.anthropic.com/news/claude-opus-4-6
- **Eval setup**: OSWorld (multimodal computer-use benchmark) and WebArena. Claude Opus 4.5 evaluated with 64k thinking budget, interleaved scratchpads, 200k context, high effort, default sampling. Reported as average of 5 independent runs.
- **Public numbers reported**: Claude Opus 4.5 achieves 66.26% on OSWorld. State-of-the-art on WebArena as single-agent system (exact score not provided in public release).
- **Methodology choices worth flagging**:
  - Uses vision capability for visual feedback loops (screenshot-based action verification)
  - Thinking budget allocation (64k tokens) balances reasoning cost vs. action speed
  - Multi-agent configurations tested (Opus 4.5 + Haiku 4.5 subagents) show 12.2% improvement over Opus 4.5 alone
- **Limitations they acknowledge**: Computer-use benchmarks are synthetic task-based; real deployment use-case distribution may differ. Long inference latency with thinking models.
- **Framework-agnostic transferability (2/5)**: Highly dependent on Computer Use API and vision modality. Requires agentic loop with screenshot feedback, tool affordance parsing. Limited portability to systems without vision or computer-control capabilities.

## OpenAI — o3/o4-mini Agentic Evaluation and ARC-AGI (2026)

### OpenAI — o3 and o4-mini Reasoning Models on Agentic Tasks (2026)
- **Source URL**: https://openai.com/index/introducing-o3-and-o4-mini/; https://evaluations.metr.org/openai-o3-report/ (redirects to https://metr.org/evaluations/openai-o3-report/)
- **Eval setup**: Agentic scaffolds adapted for reasoning models via experimentation on development task suites. o3/o4-mini use o1-adapted XML tag scaffolding (vs. Anthropic's string-based scaffolding for other models). Resource constraints: 2M token budget for baseline agents, 16M tokens for reasoning models on HCAST tasks. WebArena evaluation with variable wall-clock time limits (30 min to 8 hours; o3/o4-mini evaluated at 2-hour limit only).
- **Public numbers reported**: ARC-AGI: o3 achieves 87.5%. HCAST and RE-Bench results depend on task and reasoning model variant; detailed numbers in METR report.
- **Methodology choices worth flagging**:
  - Reasoning models require re-scaffolding (XML tags vs. string-replacement); one-size-fits-all agent scaffold does not work across model families
  - Token and time budgets asymmetrically allocated to reasoning models (16M tokens vs. 2M for base scaffold)
  - Tools include web search, file analysis, Python execution, image generation, multimodal vision
- **Limitations they acknowledge**: Asymmetric evaluation setup (reasoning models given higher budget/time) makes direct comparison problematic. Long wall-clock times (up to 2 hours per task for reasoning models).
- **Framework-agnostic transferability (2/5)**: Scaffold design is OpenAI-specific (reasoning model interaction patterns, XML tags). Requires custom adaptation for third-party systems. Token/time budget asymmetry makes portability difficult.

## Cognition AI — Devin Agent and Cognition-Golden Benchmark (2025)

### Cognition — Devin Internal Benchmark and SWE-bench Adaptation (2025)
- **Source URL**: https://cognition.ai/blog/evaluating-coding-agents; https://cognition.ai/blog/swe-bench-technical-report; https://cognition.ai/blog/devin-annual-performance-review-2025
- **Eval setup**: **Cognition-Golden**: Internal benchmark with realistic, economically-valuable development tasks on large codebases (millions of lines). Train split for autonomous self-improvement; test split for quantitative evaluation. Tasks sourced via "Cursor Blame" method (tracing committed code back to original agent requests, creating natural query-ground-truth pairs). Autonomous evaluator agents verify outcomes using browser, shell, editing tools.
- **Public numbers reported**: Devin resolves 13.86% of SWE-bench issues end-to-end (prior SOTA: 1.96%). Internal cognition-golden scores not publicly disclosed; emphasis on "correctness, speed, good communication."
- **Methodology choices worth flagging**:
  - Evaluator agents (not LLM-as-judge) autonomously verify outcomes via tool execution
  - Tasks pulled from production commits, avoiding synthetic problem construction
  - SWE-bench adapted for agentic setting (agent navigates file structure; not given pre-selected file list)
  - Train/test split design ensures numerical score increases correlate with real-world correctness
- **Limitations they acknowledge**: Prompting differences required for reasoning models (o1-preview); slower inference latency with reasoning models. Public SWE-bench evaluation does not reflect internal benchmarking sophistication.
- **Framework-agnostic transferability (3/5)**: Cognition-golden is Cognition-internal and proprietary. SWE-bench adaptation is partially portable (file navigation without pre-selection), but full methodology requires their evaluator agent scaffold.

### Cognition — Evaluating o1 Integration (2025)
- **Source URL**: https://cognition.ai/blog/evaluating-coding-agents
- **Eval setup**: Integrated OpenAI's o1-preview reasoning model into Devin agent scaffold. Evaluated on cognition-golden. Key finding: o1 requires minimal, dense prompting (less "think out loud," fewer redundant instructions). Simulated users provide real-world interaction quality feedback.
- **Public numbers reported**: o1-preview showed "striking ability to reflect and analyze" with fewer hallucinations and confident incorrectness. Latency multiple times slower than prior OpenAI releases.
- **Methodology choices worth flagging**:
  - Prompting methodology must be re-tuned per model family
  - Simulated users evaluate communication quality alongside correctness
  - Cognition-golden's use of environment signals (test pass, code execution) provides objective grounding
- **Limitations they acknowledge**: Inference latency trade-off with reasoning depth. Reduced adherence to prescriptive instructions.
- **Framework-agnostic transferability (2/5)**: Findings about o1 prompting are model-specific. Methodology depends on having access to Devin's agent loop and evaluation environment.

## Cursor — CursorBench and Online Evaluation (2026)

### Cursor — CursorBench: Production-Grounded Agent Evaluation (2026)
- **Source URL**: https://cursor.com/blog/cursorbench
- **Eval setup**: Hybrid offline-online approach. **Offline (CursorBench)**: Tasks sourced from real Cursor production sessions via "Cursor Blame" method (traces committed code to agent requests). Measures four dimensions: solution correctness, code quality, efficiency, interaction behavior. Agentic graders (not rigid rubrics) score tasks. Plots correctness vs. median completion tokens to capture compute/latency tradeoff. **Online**: Tracks high-level proxies of agent outcomes (interaction + output quality signals) over live user sessions.
- **Public numbers reported**: CursorBench produces wider separation between models at frontier levels (where public benchmarks saturate). Rankings correlate closely with developer experience vs. public benchmark rankings.
- **Methodology choices worth flagging**:
  - Task descriptions intentionally short (vs. detailed GitHub issues in SWE-bench), mimicking real developer queries
  - Agentic graders allow multiple valid solutions (vs. public benchmarks penalizing alternative correct approaches)
  - Cursor Blame provides automatic query-solution pairing, avoiding synthetic problem construction
  - Online metrics validate that offline improvements translate to user experience
- **Limitations they acknowledge**: SWE-bench Verified had ~60% flawed tests (OpenAI discovery); CursorBench avoids this by using only committed, merged solutions. Short task descriptions may miss edge-case intent in developer queries.
- **Framework-agnostic transferability (1/5)**: CursorBench is tightly coupled to Cursor's production codebase, IDE telemetry (Cursor Blame), and agentic grader architecture. Not portable to third-party systems.

## Google DeepMind — Gemini Model Cards and Agentic Evals (2025–2026)

### Google DeepMind — Gemini Evaluation Methodology (2025–2026)
- **Source URL**: https://deepmind.google/models/evals-methodology/gemini-3-flash; https://storage.googleapis.com/deepmind-media/Model-Cards/Gemini-3-Pro-Model-Card.pdf; https://deepmind.google/models/model-cards/gemini-3-1-pro/
- **Eval setup**: Multi-benchmark evaluation across reasoning, multimodal, agentic tool use, multilingual, long-context, and safety domains. For agentic tool use: **DeepSearchQA** benchmark (900 hand-crafted "causal chain" tasks across 17 fields; each step depends on prior analysis). Pass@8 vs. pass@1 evaluation shows value of parallel trajectory exploration. Automated evaluations (not human eval); manual red teaming by specialist teams outside model team. Frontier Safety Framework covers CBRN, cyber, harmful manipulation, ML R&D, and misalignment.
- **Public numbers reported**: Gemini 3.1 Pro outperforms next-best model on frontier reasoning benchmarks. Specific agentic scores not uniformly reported in public releases; emphasis on model card transparency over single leaderboard numbers.
- **Methodology choices worth flagging**:
  - Causal chain design ensures tasks require multi-step reasoning (not single-turn)
  - Pass@K evaluation (multiple trajectories) quantifies reasoning exploration breadth
  - Safety evaluation separate from capability evaluation; specialist teams independent of development
  - Results sourced from official provider APIs for non-Google models (ensuring consistency)
- **Limitations they acknowledge**: Model cards acknowledge safety evaluation is automated, not human-verified. Long-context evaluation setup not fully disclosed. Evaluations may not cover novel use-cases outside training distribution.
- **Framework-agnostic transferability (2/5)**: Gemini model cards describe their internal evaluation methodology, but DeepSearchQA and evaluation infrastructure are Google-internal. Multi-benchmark approach is conceptually portable; hand-crafted task construction limits adoption.

## Industry Methodology Patterns

Across six major organizations (Anthropic, OpenAI, Cognition, Cursor, Google DeepMind), common evaluation patterns emerge for deployed agentic AI systems:

**1. Production-grounded benchmarking** is increasingly preferred over synthetic tasks. Cursor's Cursor Blame, Cognition's traced commits, and Anthropic's SWE-bench (real GitHub issues) all prioritize authentic task distribution over constructed problems. This reflects recognition that synthetic benchmarks suffer from contamination and misalignment with real developer/user needs.

**2. Agentic grading supersedes rigid rubrics.** Both Cursor (agentic graders) and Cognition (evaluator agents) move away from LLM-as-judge or binary-outcome metrics. Evaluator agents execute tasks in live environments, providing objective signal (test pass, code compilation, human feedback) rather than probabilistic judgment.

**3. Multi-dimensional scoring** (correctness + efficiency + interaction quality) captures value tradeoffs. Cursor's token-efficiency plots and Cognition's communication scoring recognize that frontier models present latency/accuracy/cost tradeoffs not reducible to pass/fail.

**4. Reasoning models require model-family-specific scaffolding.** OpenAI (XML tags for o1/o3) and Cognition (prompting re-tuning for o1-preview) both discovered that reasoning models break assumption that one agent scaffold fits all. This implies evaluation setup is not framework-portable across model families.

**5. Behavioral evaluation complements capability evaluation.** Anthropic's Bloom framework quantifies misalignment behaviors (sycophancy, self-preservation) separately from task completion. This reflects growing recognition that deployed systems need both capability and safety measurement.

**6. Online validation gates offline metrics.** Cursor and Cognition both measure live user impact (developer satisfaction, faster code resolution) to validate that benchmark improvements translate to real-world outcomes. Offline benchmarks alone risk optimizing for benchmark artifacts.

Common limitation: **Evaluation methodology asymmetry across model families** makes cross-system comparison difficult. Cognition and OpenAI acknowledge that reasoning models require higher token/time budgets, different scaffolding, and different prompting—yet publication practices often report single headline numbers (e.g., "o3 achieves 87.5% on ARC-AGI") without disclosing these asymmetries. This risks misinterpreting benchmark rankings as model-family-neutral comparisons.
