# S1.5: Real-World Deployment Evaluation Cases

## Cognition Devin — Annual Performance Review (2025)

- **Source URL**: https://cognition.ai/blog/devin-annual-performance-review-2025
- **Author affiliation**: Vendor (Cognition Labs)
- **What was evaluated**: Real-world deployment across 18 months; PR merge rates, vulnerability resolution speed, code migration tasks, problem-solving time, resource efficiency
- **Method**: Live production telemetry; 67% PR merge rate vs 34% previous year; cost per task metrics for vulnerability patching, ETL migration, Java upgrades
- **Reported numbers**: 4x faster at problem-solving; 2x more efficient resource consumption; 67% PR merge rate; 20x efficiency gain on vulnerability resolution (30 min → 1.5 min per task); 10x improvement on ETL migrations; 14x faster Java upgrades
- **Critical observations**:
  - Struggles with ambiguous requirements and mid-task requirement changes; works best with upfront scoping
  - Cannot manage people, emotions, or handle stakeholder interactions
  - Success depends heavily on how well engineers "manage" Devin (clear specifications required)
- **Reproducibility (2)**: Closed metrics; vendor-reported telemetry only; no third-party audit trail
- **Framework-agnostic insight**: Real-world deployment metrics (PR merge rate, velocity improvements) reveal agent effectiveness differs sharply from benchmark scores; qualitative failure modes (ambiguity intolerance, inability to self-scope) matter more than raw task completion rates.

### Cognition Devin — SWE-bench and Critique (2024–2025)

- **Source URL**: https://cognition.ai/blog/swe-bench-technical-report; https://github.com/CognitionAI/devin-swebench-results
- **Author affiliation**: Vendor + third-party retrospectives (The Register, Trickle blog)
- **What was evaluated**: SWE-bench subset (570 issues, 25% of full test set); real-world task completion from independent reviewers
- **Method**: Benchmark (79/570 issues resolved = 13.86%); independent user testing (real 5-day trial, 20-task assignment, real codebase scenarios)
- **Reported numbers**: 13.86% SWE-bench pass rate (vendor); 14–15% real-world success from third-party testing; 3/20 clear successes, 14 failures, 3 unclear in one study; failures include infinite loops on recursion, integration conflicts, unclear decision-making
- **Critical observations**:
  - Vast gap between benchmark (13.86%) and vendor-reported real-world success (67% PR merge); gap explained by selection bias (merged PRs are easier than full issue resolution)
  - Struggles with recursive logic, third-party library conflicts, ambiguous specifications
  - Excels at bounded, clear-scoped tasks (junior-level 4–8 hour work)
- **Reproducibility (3)**: Benchmark reproducible (SWE-bench public); third-party retrospectives also available; vendor telemetry closed
- **Framework-agnostic insight**: Publishing benchmark scores without contextualizing to real-world success rates creates perception gap; task selection bias (easy PRs vs full issues) can inflate apparent effectiveness by 4–5x.

## GitHub Copilot — Evaluation Framework and Methodology (2024)

- **Source URL**: https://github.blog/ai-and-ml/generative-ai/how-we-evaluate-models-for-github-copilot/
- **Author affiliation**: Vendor (GitHub)
- **What was evaluated**: Code completion quality, bug-fixing ability, chat answer accuracy, language distribution (Java 57% correctness vs JavaScript 27%)
- **Method**: Offline testing (4,000+ automated tests); containerized repositories with deliberately broken tests; live internal canary testing; multi-language evaluation
- **Reported numbers**: Java 57% correctness, JavaScript 27%; 50% time saved on documentation, 30–40% on repetitive coding/testing/debugging; hard problems show 40% success across Copilot and ChatGPT
- **Critical observations**:
  - Large language variation in model correctness suggests context/idiom sensitivity
  - Live testing required before public release; offline tests insufficient for real-world validation
  - Hard problem success rate (40%) indicates ceiling even for frontier models
- **Reproducibility (3)**: Methodology public; 4,000+ test suite not open; live eval results not shared per-model
- **Framework-agnostic insight**: Multi-layer evaluation (offline → canary → production) catches model-to-language brittleness; publishing aggregate success rates masks skill distribution across task categories.

## GitHub Copilot Workspace — Agent Mode Evaluation (July 2024)

- **Source URL**: https://github.blog/ai-and-ml/github-copilot/agent-mode-101-all-about-github-copilots-powerful-mode/; https://epiverse-trace.github.io/posts/copilot-workspace/
- **Author affiliation**: Vendor (GitHub) + third-party field report (Epiverse-TRACE)
- **What was evaluated**: Autonomous repository research, implementation planning, code change execution, documentation understanding
- **Method**: Iterative agent loop (context determination → code edit + terminal commands → correctness monitoring); real-world package documentation test (R Roxygen tags)
- **Reported numbers**: (insufficient public benchmark data); partial success on R documentation linking (correctly modified export documentation but no mechanistic understanding of doc-generation workflow)
- **Critical observations**:
  - Agent does not possess task-specific domain knowledge (e.g., R package development conventions)
  - Automated testing in loop helps catch errors but does not guarantee mechanistic understanding
  - Back-and-forth between prompt tuning and real-world validation ongoing; different behavior across GPT-4o vs Claude Sonnet
- **Reproducibility (2)**: No published benchmark suite; agent behavior evaluation required custom domain knowledge
- **Framework-agnostic insight**: Black-box agent evaluation on unfamiliar domains reveals whether agent has acquired task semantics or merely learned syntactic patterns; partial success on surface tasks masks deep understanding gaps.

## Cursor — Difference-in-Differences Study (2024–2025)

- **Source URL**: https://arxiv.org/html/2511.04427v2
- **Author affiliation**: Third-party (empirical software engineering study)
- **What was evaluated**: Impact of Cursor AI on real repository velocity, code quality, developer behavior across 807 repositories (adoption Jan 2024 – Mar 2025)
- **Method**: Difference-in-differences design; treated/control repositories; real development workflows; longitudinal adoption tracking
- **Reported numbers**: Users report 8–12 hours/week saved on multi-file projects; agent usage grew >15x over past year; 2:1 ratio (agent users >> autocomplete users by early 2026); adoption: 1M+ daily users, 50,000 businesses
- **Critical observations**:
  - Stability degrades on large monorepos; context loss in heavily modular architectures common
  - Complex architectural decisions and domain-specific logic trip up agent
  - Privacy Mode restricts background agent and memory features
- **Reproducibility (4)**: Empirical study published; 807 repository sample; methodology replicable; vendor telemetry (usage counts) proprietary
- **Framework-agnostic insight**: Longitudinal adoption data (agent >> autocomplete) reveals developer preference shift; real codebase size/complexity emerges as key moderator of effectiveness, not visible in controlled benchmarks.

## METR — Experienced Developer AI Tool Impact Study (Early 2025)

- **Source URL**: https://metr.org/blog/2025-07-10-early-2025-ai-experienced-os-dev-study/
- **Author affiliation**: Third-party (METR research organization)
- **What was evaluated**: AI tool impact (Cursor Pro with Claude 3.5/3.7 Sonnet) on 246 real open-source issues; experienced developers; randomized issue assignment
- **Method**: Randomized controlled trial; 16 developers from major repos; paired issues (AI allowed vs disallowed); identical task types; time-to-completion, code quality, developer expectation vs reality
- **Reported numbers**: **19% slowdown** when AI tools available (contradicting developer expectation of 24% speedup); developers expected 24% speedup, reported perceived 20% speedup despite measured slowdown; same code quality submitted regardless of AI availability
- **Critical observations**:
  - Counterintuitive slowdown likely due to five identified factors (tool friction, validation overhead, distraction, context-switching, false confidence in incorrect code)
  - Large perception-reality gap: developers *believe* AI helps despite objective slowdown
  - Snapshot of early-2025 frontier models; findings may not hold as AI systems improve
- **Reproducibility (5)**: Full methodology published; randomized controlled design; reproducible setup; no proprietary data
- **Framework-agnostic insight**: Measuring elapsed time vs perceived productivity reveals assumption bias in developer self-assessment; black-box tools introduce invisible friction (validation loops, context fragmentation) that benchmarks cannot capture.

## Aider — Code Editing Benchmark and Leaderboard (2024–2025)

- **Source URL**: https://aider.chat/docs/benchmarks.html; https://aider.chat/docs/leaderboards/edit.html; https://aider.chat/2024/12/21/polyglot.html
- **Author affiliation**: Vendor (Aider), third-party hosting (llm-stats.com, Epoch AI)
- **What was evaluated**: LLM code editing capability (Python 133 exercises → Polyglot 225 exercises across C++, Go, Java, JS, Python, Rust); model ability to translate natural language to tests-passing code; error correction from test feedback
- **Method**: Two-attempt benchmark; first attempt graded, feedback given, second attempt graded; unit-test-pass criterion; Exercism-derived exercises
- **Reported numbers**: Leaderboard leaders (GPT-5: 88%, Gemini 2.5 Pro: 82.2%, o3: 81.3% on polyglot); polyglot benchmark intentionally hard (only solved by 3 or fewer models per problem); failure rates typically 30–50% on polyglot
- **Critical observations**:
  - High variance in performance across languages; JavaScript/newer langs show lower success rates
  - Second-attempt correction rate indicates error diagnosis ability; models vary widely
  - Problem selection (only 3-or-fewer solutions) creates ceiling effect for weaker models
- **Reproducibility (4)**: Benchmark tasks public; leaderboard accessible; exercise set open-source; allows third-party reproduction and ranking verification
- **Framework-agnostic insight**: Two-attempt design (initial + error feedback) measures both problem-solving and debugging; language-specific performance variation reveals model training data bias, not generalizable reasoning.

## Replit Agent — Deployment Metrics (2024–2025)

- **Source URL**: https://blog.replit.com/2025-replit-in-review
- **Author affiliation**: Vendor (Replit)
- **What was evaluated**: Code generation volume, deployment success, test coverage, bug fix automation, response latency
- **Method**: Production telemetry (lines of code generated, build times, deployment success rates, test coverage improvements, uptime)
- **Reported numbers**: 2.5M lines generated daily average; 12-min average app build time; 92% first-try deployment success; 87% automatic bug fixes; 65% test coverage boost; 76% reduction in code review time; 2.3-sec response latency; 98.7% uptime; <4% production hallucination rate; 500K unique users assisted; 1M prototypes in first 90 days; 300% MoM user growth (Aug 2024)
- **Critical observations**:
  - Company-reported metrics lack third-party audit; "4% hallucination" is self-measured and may undercount
  - <4% production hallucination seems unrealistically low for frontier models on real code
  - User growth metrics (300% MoM) suggest strong product-market fit but do not validate capability claims
- **Reproducibility (1)**: No independent verification; vendor telemetry only; no public benchmark or task set
- **Framework-agnostic insight**: Vendor-reported "no hallucination" claims require skepticism absent third-party spot-check; production metrics (uptime, latency) are easier to verify than capability metrics (bug fix correctness, test coverage improvement).

## v0 by Vercel — Code Generation Benchmark (2024–2025)

- **Source URL**: https://vercel.com/blog/v0-composite-model-family; https://fireworks.ai/blog/vercel
- **Author affiliation**: Vendor (Vercel) + inference partner (Fireworks AI)
- **What was evaluated**: React/Next.js UI code generation; error-free generation rate; end-to-end latency; SWE-bench-adjacent task distribution
- **Method**: Composite model family (retrieval-grounded + frontier LLM reasoning + streaming AutoFix post-processor); latency and error-free generation benchmarked via Fireworks RFT + speculative decoding; compared to Claude Sonnet 3.5 baseline (62% compilation rate)
- **Reported numbers**: 93rd percentile error-free generation rate (implies ~93% on internal benchmark); 40x improvement in end-to-end latency; Claude Sonnet 3.5 baseline 62% compilation; v0 models outperform base counterparts but larger model (v0-1.5-lg) shows higher error rate than smaller
- **Critical observations**:
  - Domain-specific (React/UI) tool; not general-purpose code generation; claims not transferable to other languages
  - "Error-free generation" metric unclear (syntax only? semantic? does it pass all tests?)
  - AutoFix post-processor inflates apparent quality by catching and fixing errors mid-stream
- **Reproducibility (2)**: Benchmark internal; baseline comparison (Claude Sonnet 62%) reproducible but full v0 eval not public; custom metrics
- **Framework-agnostic insight**: Specialized tools (UI only) benchmark against generalist models (Sonnet), creating misleading comparison; post-processing (AutoFix) moves quality measurement from model to pipeline, obscuring where errors originate.

## Bolt.new — Code Generation Case Study (2024)

- **Source URL**: https://trickle.so/blog/bolt-new-review; https://github.com/stackblitz/bolt.new
- **Author affiliation**: Third-party review (Trickle blog)
- **What was evaluated**: Full-stack app generation (React, Node.js, PostgreSQL) from natural language; debugging capability; code modularity; deployment experience
- **Method**: Real-world project trials; single-prompt generation; iterative refinement; reported time-to-failure and token usage on realistic tasks
- **Reported numbers**: Typical project: 20M+ tokens consumed debugging single auth issue; one-click deployment works for MVP; code duplication and lost pattern consistency on projects >5K LOC; debugging limited (spots syntax, misses architecture)
- **Critical observations**:
  - Full-stack generation sounds impressive but real-world use shows immature debugging (misdiagnoses architectural issues, catches only syntax)
  - Token efficiency extremely poor for real projects; 20M tokens on one issue suggests cost-prohibitive iteration loop
  - Code quality degrades non-linearly with project size; modularity breaks at scale
- **Reproducibility (3)**: Real-world trial; single anecdotal case; trial methodology not standardized; no quantitative gate
- **Framework-agnostic insight**: Full-stack tools hide fundamental debugging gap (syntax vs architecture); code duplication emergence as projects scale suggests models lack long-context coherence on generated code.

## Cline / Roo Code — Comparative Framework Discussion (2024–2025)

- **Source URL**: https://github.com/cline/cline/issues/9174; https://medium.com/@haveainiceday/code-acceleration-showdown-roo-code-vs-cline-who-wins-your-workflow-38ae48e3a5a4
- **Author affiliation**: Community (Cline GitHub issue) + third-party blog (Medium)
- **What was evaluated**: Multi-file refactoring, speed (React component generation), autonomy/control tradeoff, customization capability
- **Method**: Anecdotal side-by-side comparison; speed test (React component: Cursor 45s, Copilot 60s, Cline 90s); file detection accuracy (Cline 7/8, Cursor 8/8); control/autonomy philosophy review
- **Reported numbers**: Cline 2x slower than Cursor (90s vs 45s); multi-file detection 7/8 vs Cursor 8/8; Roo Code fork of Cline developed faster feature velocity
- **Critical observations**:
  - Cline emphasizes control/transparency; Roo Code emphasizes autonomy/speed (reflecting fork philosophy)
  - Latency differences (2x) likely due to architecture (stepwise planning vs role-based agents)
  - Community discussion, not published benchmark; results anecdotal
- **Reproducibility (2)**: Anecdotal community comparison; no standardized test harness; each trial likely uses different prompts/models
- **Framework-agnostic insight**: Control vs autonomy is design tradeoff visible in latency (Cline slower but visible reasoning) vs speed (Roo faster but black-box agents); no unified framework exists to evaluate both simultaneously.

## Cross-Case Patterns

**What gets measured well:**
- Latency and throughput (uptime, lines generated, response time) — objectively observable, hard to game
- Benchmark task pass rates (SWE-bench, Aider, Exercism) — reproducible, comparable across models
- Real-world adoption metrics (user growth, PR merge rates, velocity improvements) — indicate product-market fit but mask capability details
- Error types (architecture vs syntax, ambiguity handling, recursion) — reveal model blindspots not visible in aggregate scores

**What gets glossed over:**
- Perception-reality gap (19% slowdown measured vs 20% speedup perceived; Devin benchmark 13.86% vs implied real-world 67% success) — suggests developers conflate velocity feel with actual productivity
- Domain-specific brittleness (Copilot JavaScript 27% vs Java 57%; v0 only UI; Bolt.new hidden token cost per issue) — vendor claims rarely contextualize to specific domains
- Post-processing inflation (v0 AutoFix, Aider two-attempt design, Devin PR selection bias) — pipeline components move quality credit away from core model
- Failure mode distribution (when does agent give up? recurse infinitely? choose wrong architecture?) — reported as rare ("4% hallucination") or absent in most vendor reports
- Reproducibility variance (METR 5/5 vs Replit 1/5 vs Aider 4/5) — no consensus benchmark across industry, making cross-tool comparisons unreliable
